# Dual-Format SBOM Data-Placement Map (CycloneDX 1.6 ↔ SPDX 2.3 ↔ SPDX 3.0.1)

**Version**: v1 | **Canonical home**: this file (`docs/reference/sbom-format-mapping.md`). A mirror copy lives at `specs/010-spdx-output-support/contracts/sbom-format-mapping.md` for plan-review purposes; when the two disagree, *this file wins*.

This document is the contract that the SPDX 2.3 serializer (and the SPDX 3 stub) MUST honor. Per FR-013, FR-014, FR-015, every data element mikebom emits in CycloneDX has a row here naming its target location in each format and a one-line justification.

## Conventions

- **CycloneDX location**: JSON Pointer-style path into the CycloneDX 1.6 document (`/components/{i}/<field>` etc.).
- **SPDX 2.3 location**: Either a JSON path into the SPDX 2.3 document (`/packages/{i}/<field>`), an SPDX `Relationship` description, an SPDX `Annotation` with the named `field` envelope (per `mikebom-annotation.schema.json`), or `omitted — <reason>`.
- **SPDX 3.0.1 location**: Either a JSON-LD path into a 3.0.1 element graph (`{Element}/<property>`), an `Annotation` element pointer, an `ExternalRef` / `ExternalIdentifier` entry shape, or `omitted — <reason>` for entries mikebom intentionally does not emit.
- "**Annotation** ⟨field⟩" is shorthand for: emit one `SpdxAnnotation` whose `comment` is the JSON-encoded `MikebomAnnotationCommentV1` envelope with the named `field` value.

## Section B — Graph structure (relationships)

| # | mikebom data | CycloneDX 1.6 location | SPDX 2.3 location | SPDX 3.0.1 location | Justification |
|---|--------------|------------------------|-------------------|---------------------|---------------|
| B1 | dependency edge (runtime) | `/dependencies[]/dependsOn[]` | `Relationship` with `relationshipType: "DEPENDS_ON"` | `Relationship` element with `relationshipType: "dependsOn"` | Native graph edges in all. |
| B2 | dependency edge (dev/build/test scope) | `/components/{i}/scope = "excluded"` (native CDX 1.6 enum value, set on the dep target) **plus** `mikebom:lifecycle-scope` property per C42 carrying the finer `development` / `build` / `test` distinction | **Default (`--spdx2-relationship-compat=full`):** native `Relationship` `relationshipType` of `DEV_DEPENDENCY_OF` / `BUILD_DEPENDENCY_OF` / `TEST_DEPENDENCY_OF` (one per scope value, reversed direction per spec) — the SPDX 2.3 spec's purpose-built field for the dev/build/test distinction. **`--spdx2-relationship-compat=basic`:** flat `DEPENDS_ON` edges in natural direction; scope information lives only on the target Package's `mikebom:lifecycle-scope` annotation per C42. **In both modes** the target Package also carries the `mikebom:lifecycle-scope` annotation so consumers walking `DEPENDS_ON` only — the Trivy / Syft convention — can still distinguish dev/build/test deps from deployed-runtime deps. **Both modes are spec-conformant but NOT equivalent**: `full` is the spec-native expression and the mikebom default per Constitution Principle X (Transparency); `basic` is a deliberate downshift for tool-compat reach. **This distinction is consumer-critical signal**: vulnerability scanners, license auditors, container deployment policies, and SBOM-diff tools all need to know which components ship to production vs which are dev-only or test-only. | Native `Relationship` `relationshipType: "dependsOn"` **plus** `scope` field on the relationship element carrying the SPDX 3 `LifecycleScopeType` enum value (`development` / `build` / `test`). Runtime-scoped edges omit `scope`. | Milestone 052 redesign: pre-052 used a custom `mikebom:dev-dependency` annotation in all three formats. Per Constitution Principle V (v1.4.0), every format has native fields for this signal — they're now used as the primary mechanism. CDX 1.6's 3-value `scope` enum (`required`/`optional`/`excluded`) cannot express the dev-vs-build-vs-test split, so the finer signal lives in `mikebom:lifecycle-scope` per C42 (Principle V finer-info carve-out). Issue #228 extended C42 to SPDX 2.3 as a parity-bridging annotation (Principle V parity-gap carve-out) so the dev/build/test distinction is recoverable from a `DEPENDS_ON`-walk under either relationship-compat mode. |
| B3 | nested containment (shade-jar parent → children) | `/components/{i}/components[]` (CDX nests inline) | Flat `Package`s + `Relationship` with `relationshipType: "CONTAINS"` from parent SPDXID to child SPDXID | Flat `Package`s + `Relationship` `relationshipType: "contains"` | SPDX has no nesting; containment is expressed by relationships. Edge case: nested components. |
| B4 | image / filesystem root | `/metadata/component` | `documentDescribes: [<root SPDXID>]` plus `Relationship` `DESCRIBES` from document to root | `SpdxDocument/rootElement` | All three formats name the root differently; mapping is mechanical. Synthesize a root if no natural one exists. |

## Section H — Structural differences between formats

Rows here document format-level representational differences that don't fit cleanly in per-row mappings. They're NOT bugs; they're spec-level differences that consumers comparing mikebom outputs across formats need to be aware of.

| # | Topic | CycloneDX 1.6 | SPDX 2.3 | SPDX 3.0.1 | Justification |
|---|-------|---------------|----------|------------|---------------|
| H1 | Nested vs. flat component representation | **Nested** via `component.components[]` (CDX 1.6 §6.2.10) — e.g. Maven shade-plugin fat-jars emit vendored coords nested under the fat-jar top-level component. Mikebom's `cyclonedx/builder.rs::build_components` filters `parent_purl.is_none()` for top-level emission and folds children via `effective_parent`. | **Flat** `packages[]` (SPDX 2.3 §7 — spec forbids nesting). Parent-child containment expressed via `Relationship[relationshipType: "CONTAINS"]` edges. Every `ResolvedComponent` becomes a top-level Package regardless of `parent_purl`. | **Flat** `@graph` `software_Package` elements. Same pattern as SPDX 2.3: parent-child via `Relationship[relationshipType: "contains"]`. | Consequence: CDX's `cdx.components.length` counts *top-level only*; SPDX 2.3's `spdx23.packages.length` counts *everything*. Comparing the two raw counts will drift by the count of nested children. The [`component_count_parity.rs`](../../mikebom-cli/tests/component_count_parity.rs) test locks the invariant: `cdx.flattened_count == spdx23.packages.length - synthetic_root_count == spdx3.software_Package.count - synthetic_root_count`. The user-reported "22-component drift" on the external `polyglot-builder-image` fixture came from comparing top-level CDX to flat SPDX — not a bug, a format-level difference. See `specs/012-sbom-quality-fixes/research.md` §R2 for the full investigation trace. |
