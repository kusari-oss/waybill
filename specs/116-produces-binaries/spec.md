# Feature Specification: Automatic binary-name binding via source-tier `produces-binaries` annotation

**Feature Branch**: `116-produces-binaries`
**Created**: 2026-06-13
**Status**: Draft
**Input**: User description: "Option B of issue #225 — source-tier SBOMs emit a `produces-binaries` annotation on the main-module component listing the expected binary names the build produces. The cross-tier `--bind-to-source` flow then automatically aliases `pkg:generic/<name>` image components to source-tier ecosystem PURLs without operator-supplied `--pkg-alias` flags. Extends milestone 111 (which shipped Option A of issue #225)."

## Clarifications

### Session 2026-06-13

- Q: For Cargo, which manifest/filesystem sources does the extractor read, and where do declarations land in a workspace? → A: `[[bin]]` + `src/main.rs` + `src/bin/*.rs` (Cargo's full three-source binary convention) from each member crate; declarations land on every workspace member's main-module component, not consolidated onto the workspace-root component.
- Q: For Maven, how does the JVM `.jar` extension convention bridge to image-side `pkg:generic/baz`? → A: The extractor emits the extensionless canonical name (`baz`) only; the binder's FR-002 suffix-tolerance does the `.jar` bridging. Every ecosystem emits canonical extensionless names; the binder owns all platform-suffix translation.
- Q: Which source-tier components carry the produces-binaries declaration? → A: Main-module components only — the project being scanned. Transitive dependencies do NOT carry the declaration even when their manifest data is locally readable. This matches the milestone-064/066/068/069/070 "main module is the operator's project" framing and confines the extractor scope to one component per ecosystem per scan.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — A Rust operator binds image-to-source without per-image alias flags (Priority: P1)

An operator manages a Rust project at `github.com/foo/bar` whose `Cargo.toml` declares `name = "baz"` at version `1.0.0` and produces a single binary named `baz` (the default-binary case — `src/main.rs` exists, no explicit `[[bin]]` table). CI publishes the source-tier SBOM as a release artifact, builds a container image containing the compiled `baz` binary, and emits an image-tier SBOM with `--bind-to-source <published-source-sbom>`. Today, the operator must supply `--pkg-alias "pkg:generic/baz=pkg:cargo/baz@1.0.0"` on every image scan or the flagship `baz` component is reported as `Unknown { source-not-found-in-bind-target }` because `pkg:generic/baz` ≠ `pkg:cargo/baz@1.0.0`. After this feature, the source-tier SBOM declares (on the `pkg:cargo/baz@1.0.0` component) that `baz` is one of the binary names it produces; the image-tier scan reads that declaration during `--bind-to-source` processing and treats `pkg:generic/baz` as an automatic alias for `pkg:cargo/baz@1.0.0`. The operator's image-scan command stays free of `--pkg-alias` flags, and the binding strength reports `verified` (when hash evidence aligns) or `weak` (when it doesn't), never `Unknown { source-not-found-in-bind-target }`.

**Why this priority**: This IS the textbook workflow that motivates issue #225. Option A (milestone 111's `--pkg-alias` flag) requires the operator to remember the alias for every image scan; Option B removes that burden entirely for the common case. P1 because the issue's "fully automatic for the common case" promise is only delivered by US1 — without it, the polished default the recommendation pointed at doesn't exist.

**Independent Test**: A small Rust project produces a source-tier SBOM whose main-module component carries the produces-binaries declaration naming `baz`. A containerized version of the same compiled binary is scanned with `--bind-to-source`. The image-tier SBOM's binding result for the `baz` component reports `verified` or `weak` strength, never `Unknown { source-not-found-in-bind-target }`, without any `--pkg-alias` flag on the image scan.

**Acceptance Scenarios**:

1. **Given** a Rust project whose `Cargo.toml` declares one default binary named `baz` at version `1.0.0`, **When** mikebom scans the project at source-tier (`mikebom sbom scan --path .`), **Then** the emitted SBOM's main-module component (PURL `pkg:cargo/baz@1.0.0`) carries a machine-readable declaration listing `baz` as one of its produced binary names.
2. **Given** the source-tier SBOM from scenario 1 AND a container image containing the compiled `baz` binary, **When** the operator runs `mikebom sbom scan --image <ref> --bind-to-source baz-source.cdx.json --output baz-image.cdx.json` with NO `--pkg-alias` flag, **Then** the binding result for the binary component reports `verified` or `weak` strength, the reason field is not `source-not-found-in-bind-target`, AND the emitted SBOM records that the binding was reached via the automatic alias (so a future auditor can see how the match was made).
3. **Given** the same scenario 2, **When** the operator supplies `--pkg-alias` for the SAME pair the automatic alias would also produce, **Then** the operator-supplied alias takes precedence, the automatic alias is NOT also applied, and the binding-result audit trail records that the alias source was operator-supplied.
4. **Given** a source-tier SBOM whose main-module component declares MULTIPLE produced binaries (e.g., `baz` plus `baz-cli` plus `baz-debug`), **When** the image-tier scan finds an image binary whose name matches any one of the declared names, **Then** that single image-tier binary binds to the source-tier main-module component; the un-matched declared binary names produce no binding noise.
5. **Given** a Rust project whose `Cargo.toml` declares NO binaries (it's a library crate — `lib.rs`, no `[[bin]]`, no `src/main.rs`), **When** mikebom scans it at source-tier, **Then** the emitted SBOM contains NO produces-binaries declaration on the main-module component (the library is not expected to produce any binaries — the absence of the annotation is correct, not a gap).

### User Story 2 — A polyglot operator binds image-to-source across npm, Python, Ruby, and Maven projects without per-image alias flags (Priority: P2)

The same workflow as US1, but for ecosystems beyond Cargo. An operator's organization has Rust binaries, Node.js CLIs, Python scripts, Ruby gems, and Java JARs all shipping as binary tier artifacts inside container images. They want the `--bind-to-source` automatic-alias behavior to work for each ecosystem's source-tier scan output, not just Cargo's. For each ecosystem, the source-tier scan reads the ecosystem-conventional declaration of "this project produces these executable names" — `package.json`'s `bin` field for npm, `[project.scripts]` for pip-installed projects, the gemspec's `executables` array for gem, and the maven shade-/jar-plugin's `<finalName>` declaration for jar-style executable artifacts — and emits the produces-binaries declaration on the main-module component.

**Why this priority**: The Rust example in the issue body is one of four named ecosystems (Cargo, npm, Maven, plus pip and gem implied by the milestone-064/066/068/069/070 main-module rollout pattern). Polyglot operators are common, but Cargo gets the first slice because it's the issue body's canonical case. P2 because shipping US1 alone closes the issue body's named scenario; US2 expands coverage to the rest of the workspace's main-module ecosystems.

**Independent Test**: For each ecosystem in scope (npm, pip, gem, maven), a small per-ecosystem fixture project exists whose source-tier SBOM declares its produced binary names, and a containerized variant's `--bind-to-source` scan produces a non-Unknown binding for the flagship component without `--pkg-alias` flags.

**Acceptance Scenarios**:

1. **Given** an npm project whose `package.json` declares `{"name": "baz", "version": "1.0.0", "bin": "./bin/baz.js"}`, **When** mikebom scans the project at source-tier, **Then** the emitted SBOM's main-module component (PURL `pkg:npm/baz@1.0.0`) declares `baz` as a produced binary name (the basename of the file path the `bin` field points at).
2. **Given** an npm project whose `package.json` declares `{"name": "baz", "version": "1.0.0", "bin": {"baz": "./cli.js", "baz-init": "./init.js"}}`, **When** mikebom scans it at source-tier, **Then** the emitted SBOM declares BOTH `baz` AND `baz-init` as produced binary names (one declaration carries a list, not a single name).
3. **Given** a Python project whose `pyproject.toml` declares `[project.scripts] baz = "baz.cli:main"`, **When** mikebom scans it at source-tier, **Then** the emitted SBOM's main-module component (PURL `pkg:pypi/baz@1.0.0`) declares `baz` as a produced binary name.
4. **Given** a Ruby gemspec whose `executables = ["baz"]` declaration is present, **When** mikebom scans it at source-tier, **Then** the emitted SBOM's main-module component (PURL `pkg:gem/baz@1.0.0`) declares `baz` as a produced binary name.
5. **Given** a Maven project whose POM configures the shade-plugin with `<finalName>baz</finalName>` to produce an executable JAR named `baz.jar`, **When** mikebom scans it at source-tier, **Then** the emitted SBOM's main-module component (PURL `pkg:maven/com.acme/baz@1.0.0`) declares `baz` (extensionless, canonical form) as a produced binary name. The image-tier `pkg:generic/baz` match succeeds via direct equality; the image-tier `pkg:generic/baz.jar` match succeeds via the binder's FR-002 `.jar`-suffix tolerance — the extractor never emits the `.jar` suffix.

### User Story 3 — A Go operator binds image-to-source for `package main` modules with no flag burden (Priority: P3)

Go's ecosystem doesn't declare binary names in `go.mod`; the binary name is determined by the directory of the `package main` source file (or `go build -o`). The operator's Go project at `github.com/foo/bar/cmd/baz` produces a binary named `baz`. Source-tier scan identifies the `package main` directory's basename as the expected binary name and emits the declaration on the source-tier main-module component (PURL `pkg:golang/github.com/foo/bar@v1.0.0` with a `?subpath=cmd/baz` qualifier per the milestone-053 main-module convention). The `--bind-to-source` flow then auto-aliases as in US1/US2.

**Why this priority**: Go is the trickiest case — there's no manifest field naming the binary; mikebom has to extract it from filesystem layout. Lower priority because it requires per-package-main filesystem walking inside the source-tier scan, increasing risk and surface area. P3 because Cargo + npm + pip + gem + maven (US1 + US2) cover the bulk of the polyglot case from the issue body, and Go users have the milestone-111 `--pkg-alias` workaround until US3 lands.

**Independent Test**: A Go project with `cmd/baz/main.go` declaring `package main` produces a source-tier SBOM whose main-module component declares `baz` as the produced binary name. A containerized variant's `--bind-to-source` scan binds the binary without `--pkg-alias` flags.

**Acceptance Scenarios**:

1. **Given** a Go project whose `cmd/baz/main.go` contains `package main`, **When** mikebom scans the project at source-tier, **Then** the emitted SBOM's main-module component (PURL `pkg:golang/github.com/foo/bar@v1.0.0`) declares `baz` as a produced binary name.
2. **Given** a Go project with TWO `package main` directories at `cmd/baz/` and `cmd/baz-helper/`, **When** mikebom scans it at source-tier, **Then** the emitted SBOM declares both `baz` and `baz-helper` as produced binary names.
3. **Given** a Go project whose root directory itself contains `package main` (the older convention without a `cmd/` layout), **When** mikebom scans it at source-tier, **Then** the emitted SBOM declares the root directory's basename (e.g., `bar` if the repo is `github.com/foo/bar`) as the produced binary name.

### Edge Cases

- What happens when the source-tier SBOM's main-module component already has a `produces-binaries` declaration from a prior tool, OR a downstream operator hand-edits the value? mikebom MUST preserve user-supplied additions and add its own discovered names alongside — never silently drop or overwrite operator declarations. The merge policy is union: the final declaration is the union of what mikebom discovered AND what the operator pre-set.
- What happens when the produces-binaries declaration names binaries that are NOT in the scanned image? Nothing — the un-matched declarations are silent. The annotation is a CAPABILITY claim ("this source could produce these names"), not an OBLIGATION ("this image must contain these"). An image scanner finding only one of the three declared binaries reports one binding; the other two declarations do not become "missing" failures.
- What happens when two different source-tier components both declare the same binary name? For example, a Rust workspace where two crates both produce `baz` (unusual but legal — `cargo build` resolves it by build order). The image-tier binder MUST surface the ambiguity rather than picking arbitrarily; binding strength is `weak` with reason `multiple-source-candidates-for-binary-name`, and both source candidates are listed in the audit trail.
- What happens when the image-tier binary's name differs from the source-tier declared name by case (`Baz` vs `baz`) or trailing extension (`baz.exe` on Windows-built images)? The match MUST be case-insensitive AND tolerate Windows-conventional `.exe` suffixes and JVM-conventional `.jar` suffixes on the image side. Source-side declarations carry the canonical (lowercase, extensionless) name.
- What happens when the operator passes both `--pkg-alias` AND the source SBOM contains a produces-binaries declaration that would produce the same automatic alias? The operator-supplied alias wins; the automatic alias is suppressed; the audit trail records that the alias source was operator-supplied (not automatic). This preserves operator override authority.
- What happens when the source-tier SBOM was produced by a non-mikebom tool (syft, trivy) that does not carry the produces-binaries declaration? The `--bind-to-source` flow degrades to milestone-072's exact-PURL match — which is the pre-feature behavior. The feature is additive to source SBOMs that carry the declaration; the absence does not block binding.
- What happens when a contributor in a future PR wants to add a new ecosystem (e.g., Elixir, Erlang)? The cross-tier binder's automatic-alias consumer code is ecosystem-agnostic (it reads the declaration off the component property; it does not care which ecosystem produced it). Only the source-tier per-ecosystem extractor needs new code. The cost of adding a new ecosystem is one extractor function + one fixture project, not a full binding-pipeline rewrite.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: A source-tier scan MUST emit, on each main-module component whose ecosystem manifest declares one or more produced executable / binary names, a machine-readable declaration listing those names. The declaration is a list of strings, in the canonical (lowercase, extensionless) form. The declaration is omitted entirely (NOT present as an empty list) when no binary names can be extracted from the ecosystem manifest. The declaration appears ONLY on main-module components (the project being scanned); transitive-dependency components MUST NOT carry the declaration even when their manifest data is locally readable — this matches the milestone-064/066/068/069/070 "main module is the operator's project" framing and confines the extractor scope to one component per ecosystem per scan.
- **FR-002**: The cross-tier binding flow (`--bind-to-source`) MUST read the produces-binaries declaration from the source-tier SBOM during binding, and MUST treat `pkg:generic/<declared-name>` image-tier components as automatic aliases for the source-tier main-module component carrying the declaration. The match is case-insensitive and tolerates Windows `.exe` and JVM `.jar` suffixes on the image-side name only.
- **FR-003**: When an automatic alias produces a binding match, the emitted image-tier SBOM MUST record that the binding was reached via the automatic alias in the same audit-trail position the milestone-111 operator-supplied alias uses. The audit trail MUST distinguish operator-supplied aliases (sourced from `--pkg-alias`) from automatic aliases (sourced from the source-tier declaration) so downstream auditors can interpret the binding provenance.
- **FR-004**: Operator-supplied `--pkg-alias` flags MUST take precedence over automatic aliases that would produce the same image-side PURL match. The automatic alias MUST be suppressed (not also applied alongside the operator one), and the audit trail MUST record the operator-supplied alias as the binding source.
- **FR-005**: For the Cargo ecosystem, the source-tier scan MUST extract produced binary names from THREE sources per Cargo's implicit-binary conventions: (a) `Cargo.toml` `[[bin]]` table entries (explicit), (b) the default-binary inference rule (`src/main.rs` produces a binary whose name is the package's `name` field), and (c) the implicit-`src/bin/*.rs` rule (every `src/bin/<name>.rs` file produces a binary named `<name>` even without a `[[bin]]` entry). For cargo workspaces, the declaration is emitted on EACH workspace member's main-module component (not consolidated onto the workspace-root component) — matching milestone-064's per-member main-module emission shape. Library-only crates (no `[[bin]]` entries, no `src/main.rs`, no `src/bin/*.rs` files) MUST emit no produces-binaries declaration on the main-module component.
- **FR-006**: For the npm ecosystem, the source-tier scan MUST extract produced binary names from `package.json`'s `bin` field. Both shape forms MUST be honored: the string form `"bin": "./bin/baz.js"` (single binary named after the package), and the object form `"bin": {"baz": "./cli.js", "baz-init": "./init.js"}` (multiple binaries with explicit names). For the string form the binary name is the package's `name` field (the file path is the IMPLEMENTATION, the package name is the binary name per npm convention).
- **FR-007**: For the Python ecosystem (pip / pyproject-toml), the source-tier scan MUST extract produced binary names from `[project.scripts]` and `[project.gui-scripts]` keys in `pyproject.toml`. Each key name is one produced binary name. Setup.cfg's `[options.entry_points]` (`console_scripts` + `gui_scripts` keys) MAY be honored as a fallback EITHER when `pyproject.toml` does not exist OR when `pyproject.toml` exists but declares no scripts — this supports legacy projects without pyproject.toml AND mid-migration projects whose pyproject.toml carries other metadata but leaves scripts in setup.cfg.
- **FR-008**: For the Ruby gem ecosystem, the source-tier scan MUST extract produced binary names from the gemspec's `executables` array. Each array element is one produced binary name.
- **FR-009**: For the Maven ecosystem, the source-tier scan MUST extract produced binary names from the POM's shade-plugin `<finalName>` configuration (when present) AND from the jar-plugin's `<finalName>` configuration. The extracted name is the canonical form WITHOUT the trailing `.jar` extension — the extractor never emits the `.jar` suffix. The binder's FR-002 `.jar`-suffix tolerance is the sole mechanism that bridges the extensionless source-side declaration to image-side `pkg:generic/baz.jar` components; this rule keeps every ecosystem's extractor uniform (canonical extensionless names) and confines all platform-suffix translation to the binder.
- **FR-010**: For the Go ecosystem, the source-tier scan MUST extract produced binary names from the filesystem layout: every directory containing a `*.go` file whose `package <name>` declaration is `package main` produces one binary whose name is that directory's basename. The standard `cmd/<name>/main.go` convention AND the root-of-repo `main.go` convention MUST both be honored.
- **FR-011**: The produces-binaries declaration MUST be encoded as a property on the source-tier main-module SBOM component using a name that complies with the project's Constitution Principle V bullet 5 (standards-native fields take precedence over `mikebom:`-prefixed properties). The spec author MUST audit CycloneDX 1.6 and SPDX 2.3 + 3.x for an existing native construct expressing "this package produces these executable names" before introducing a new `mikebom:*` property; the audit result and the chosen field name MUST be cited in the planning phase.
- **FR-012**: When the source-tier scan discovers a produces-binaries declaration that COLLIDES with one already present on the same component (for example, a hand-edited or operator-injected pre-existing declaration), the merged declaration MUST be the UNION of both lists (de-duplicated, lex-sorted). Operator declarations MUST NOT be silently overwritten or dropped.
- **FR-013**: When the cross-tier binder finds an image-tier `pkg:generic/<name>` for which TWO OR MORE source-tier components have declared the same binary name, the binding MUST be reported as `weak` with a reason field indicating the ambiguity (e.g., `multiple-source-candidates-for-binary-name`), and the audit trail MUST list all candidate source-side PURLs. The binder MUST NOT silently pick one candidate.
- **FR-014**: The feature MUST be backwards-compatible with milestone-072's existing `--bind-to-source` consumers. A source-tier SBOM produced by a pre-feature mikebom (or by a non-mikebom tool) lacking the produces-binaries declaration MUST continue to bind under the existing exact-PURL match rules. The new automatic-alias path is purely ADDITIVE — it expands the set of successful bindings without changing the result of any binding that succeeded before.
- **FR-015**: A consumer parsing the emitted SBOM MUST be able to determine the binary names a source-tier component is expected to produce without invoking mikebom — the declaration MUST be self-contained on the component (no separate lookup file, no required external query). Downstream tools that don't consume mikebom-specific extensions can ignore the property gracefully without errors.

### Key Entities

- **Produces-binaries declaration**: A machine-readable list of canonical binary names attached to a source-tier main-module SBOM component. Encoded as a property on the component using a name chosen per FR-011's constitution audit. Each entry is a lowercase string with any platform-specific suffix (`.exe`, `.jar`) stripped. The list is de-duplicated and lex-sorted.
- **Automatic alias**: A binder-internal mapping `pkg:generic/<name>` → `<source-tier-main-module-purl>` derived at bind time from the produces-binaries declaration on a source-tier component. Conceptually identical to milestone-111's operator-supplied alias but originates from the source-tier SBOM rather than from the operator's command line. Its provenance is recorded in the audit trail so consumers can tell the two sources apart.
- **Binding audit trail**: An extension of the milestone-072 binding-result property envelope (already used by milestone-111 to carry operator-supplied alias provenance) that gains a new "alias source" field with values `operator-supplied` or `automatic-from-produces-binaries`. The envelope's other fields (binding strength, layered-hash evidence, reason text) are unchanged.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: After this feature ships, the textbook Rust workflow described in issue #225 (`source SBOM → image SBOM with --bind-to-source`) produces a non-`Unknown` binding for the project's flagship binary component WITHOUT any operator-supplied `--pkg-alias` flag. The binding strength is `verified` when hash evidence aligns and `weak` when it doesn't; neither produces `Unknown { source-not-found-in-bind-target }`.
- **SC-002**: A polyglot operator scanning sources across Cargo + npm + pip + gem + Maven projects gets the same operator-flag-free automatic-binding behavior for every ecosystem in scope after US2 ships. Go gets the same behavior after US3 ships. No ecosystem-specific flag knowledge is required at image-scan time.
- **SC-003**: An auditor reading a post-feature image-tier SBOM can distinguish, for each binding result, whether the alias was operator-supplied (via `--pkg-alias`) or automatic (derived from the source-tier produces-binaries declaration). The distinction is preserved as a machine-readable field in the binding audit trail.
- **SC-004**: A consumer of source-tier SBOMs (an auditor, a downstream binding tool, an organization's SBOM-quality dashboard) can determine the binary names a source-tier component is expected to produce by reading the SBOM alone, with no recourse to mikebom-specific tooling or external lookups.
- **SC-005**: A source-tier SBOM produced by a pre-feature mikebom (or by a non-mikebom tool) that lacks the produces-binaries declaration MUST continue to behave EXACTLY as it did under milestone 072: exact-PURL match still works; the absence of the declaration produces NO false binding results, NO new errors, NO new warnings. Backwards-compatibility is total.
- **SC-006**: The mean operator-edit-count to produce a clean image binding drops from 1 (the milestone-111 `--pkg-alias` flag) to 0 (no operator-supplied alias necessary) for the common case in every supported ecosystem.

## Assumptions

- The feature's scope is the EXTRACTION (source-tier scan emits the declaration) AND the CONSUMPTION (image-tier `--bind-to-source` reads it). The source-tier scan continues to produce a single main-module component per project; this feature does not change the milestone-064/066/068/069/070/053 main-module identity logic. Only the component's property bag gains a new field.
- The image-tier binary analysis (milestone 096 — what produces the `pkg:generic/<name>` component for binaries inside images) is unchanged. The new automatic-alias machinery operates on the SAME image-tier output milestone 096 already produces; this feature does not redefine what counts as a binary component or how it's identified.
- The image-tier `pkg:generic/<name>` name segment is the binary file's basename WITHOUT path prefixes. mikebom's binary discoverer already produces this shape per milestone 096 / 104; the binder uses the same name segment for matching against source-side declarations.
- The Constitution audit in FR-011 will conclude that no existing CycloneDX 1.6 OR SPDX 2.3/3.x native field carries "list of executable names this package produces" semantics, and a new `mikebom:produces-binaries` (or similarly-named) property is justified per Principle V bullet 5 with the audit result cited in plan.md. If the audit instead surfaces a viable native field (e.g., CDX `externalReferences[type=executable]` of a sufficiently-shaped form), the spec adopts that field name; the user-visible behavior is unchanged regardless of which field name carries the declaration.
- The cross-tier binder's audit-trail extension (FR-003) extends the milestone-111 envelope (`alias_from` / `alias_to` / `applied_alias` fields) with one additional field naming the alias source. Whether that field is `alias_source: "automatic" | "operator-supplied"` or expressed via a different shape is a planning-phase choice; the user-visible distinction (operators and auditors can tell the two apart) is the contract.
- The case-insensitive + `.exe` + `.jar` suffix tolerance in FR-002 is unconditional. mikebom does not require an opt-in flag for it because the costs of OVER-matching (a `pkg:generic/Baz.exe` weakly binding to `pkg:cargo/baz@1.0.0` when the actual binary in the image is something else) are confined to `weak`-strength bindings whose semantics already convey "evidence is partial."
- Operators who want to OPT OUT of the automatic alias path (for example, because their workflow expects strictly exact-PURL matching) can suppress it by simply not using `--bind-to-source`, which is the only flow this feature affects. There is no per-component or per-scan opt-out; the absence of such opt-out is intentional — adding one would invite operators to silently suppress real bindings as workaround for unrelated problems, mirroring the spec's "no bypass mechanism" stance on the walker-audit gate (milestone 115).
- The "Go" ecosystem in US3 means modules written for the Go toolchain; the milestone-053 main-module identity logic is the canonical source of the source-tier component PURL. This feature does not redefine that logic; it only adds the produces-binaries extraction step.
- A small "negative test" — a synthetic source-tier SBOM with NO produces-binaries declaration combined with an image scan — verifies the backwards-compatibility (SC-005) acceptance criterion. A full integration test for every ecosystem (US1 + US2 + US3) verifies the per-ecosystem extraction. The cross-tier binder gets its own focused test for the alias-source distinction (FR-003).
- The maven JAR-naming convention in FR-009 (canonical form WITHOUT `.jar` suffix) means the declaration carries `baz`, not `baz.jar`. The image-side binder's `.jar`-suffix tolerance (FR-002) is the SOLE mechanism that bridges the convention to image-side `pkg:generic/baz.jar` components. The extractor never emits the `.jar` suffix; the binder owns the suffix-translation logic uniformly across all ecosystems.
