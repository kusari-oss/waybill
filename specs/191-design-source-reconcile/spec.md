# Feature Specification: Design-Tier / Source-Tier Reconciliation

**Feature Branch**: `191-design-source-reconcile`
**Created**: 2026-07-14
**Status**: Draft
**Input**: User description: "m191: bundle #558 (fix trailing-@ PURL emission on design-tier components) + #560 (reconcile design-tier and source-tier — collapse when resolved match exists, attach requirement-range as annotation on the survivor)"

## Clarifications

### Session 2026-07-14

- Q: When multiple sibling manifests declare different ranges resolving to the same source-tier component, how should the survivor encode all declared ranges? → A: Multiple `mikebom:requirement-range` properties, each paired with a distinguishing `mikebom:source-manifest` property (Option B). Format-idiomatic (matches CDX property-array + m111 pkg-alias-binding shape); preserves the range-→-manifest pairing bidirectionally.
- Q: How should the FR-001 "manifest-scope proximity" match handle workspace monorepos where the lockfile lives only at the workspace root? → A: Walk up to workspace-parent directories (Option A). A declaration in `packages/foo/package.json` matches a resolution in `<root>/package-lock.json` if the parent workspace claims `packages/foo` as a member. Reuses the existing m127 root-pick + m149 mainmod demote workspace-parentage plumbing. Strict same-directory would leave nested-declaration components unreconciled — defeating the milestone's purpose on the React Native customer shape.
- Q: What bom-ref / SPDXID / spdxId shape should standalone versionless design-tier components carry? → A: The spec-clean versionless PURL as-is (`bom-ref: "pkg:npm/commander"`), matching mikebom's existing "PURL == bom-ref" convention (Option A). Collision-avoidance is natural because source-tier IDs always carry `@version`. When a future scan resolves the dep and reconciliation applies per US1, the source-tier's `@version`-carrying ID replaces the versionless one — the shift is visible in a diff but the shape stays spec-clean throughout.
- Q: What observability should the reconciliation pass emit? → A: INFO-level summary log per scan (`reconciled N design-tier components into source-tier siblings; K standalone design-tier components emitted`) + DEBUG-level per-component logs for troubleshooting (Option B). Matches m112/m158/m173 observability level. No document-scope annotation — reconciliation is a mechanical transformation, not a diagnostic signal worth persisting in the SBOM.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Reconcile design-tier and source-tier components (Priority: P1)

An operator scans a codebase with sibling manifest + lockfile artifacts (e.g., an npm project with `package.json` + `package-lock.json`, a Cargo workspace with `Cargo.toml` + `Cargo.lock`, a Python project with `pyproject.toml` + `poetry.lock`). Today mikebom emits two near-duplicate components for each declared-and-resolved dependency: a design-tier component carrying the requirement range (`pkg:npm/commander@` with empty version + `mikebom:requirement-range: 11.1.0`) AND a source-tier component carrying the resolved concrete version (`pkg:npm/commander@11.1.0` with hashes, licenses). After m191, mikebom emits exactly ONE component per declared-and-resolved dependency — the source-tier component — with the design-tier's requirement-range and source-manifest annotations attached. Downstream SBOM consumers that dedup by PURL no longer see near-duplicates, and no information is lost.

**Why this priority**: This is the architectural fix for a customer-visible correctness gap: a real customer scan on a large React Native monorepo produced 1998 components, of which 101 were design/source near-duplicate pairs. PURL-dedup vulnerability scanners, policy engines, and SBOM diff tools all treat these as separate identities, inflating the reported surface and creating spurious signals. Fixing this at the emission layer preserves every fact (range, manifest link, dev/prod scope, resolved version, hashes, licenses) while producing cleaner, spec-idiomatic SBOMs. Filed as issue #560. Precedent: milestone-149 (`demote-manifest-mainmod`) already established the demote-when-source-match-exists pattern for main modules — this generalizes it to all declared dependencies.

**Independent Test**: Scan a fixture npm project with a manifest declaring `commander: "^11.1.0"` and a lockfile resolving it to `11.1.0`. Assert the emitted CDX 1.6 SBOM contains exactly ONE `commander` component whose PURL is `pkg:npm/commander@11.1.0`, whose `version` is `11.1.0`, and whose properties/annotations include BOTH `mikebom:requirement-range` and `mikebom:source-manifest` (the design-tier metadata carried onto the reconciled survivor). Repeat across SPDX 2.3 and SPDX 3.0.1 outputs and assert semantic equivalence.

**Acceptance Scenarios**:

1. **Given** an npm project with `commander: "^11.1.0"` declared in `package.json` and resolved to `11.1.0` in `package-lock.json`, **When** scanned with `--format cyclonedx-json`, **Then** exactly ONE `commander` component appears in the output; its PURL is `pkg:npm/commander@11.1.0` (spec-clean); its properties include `mikebom:requirement-range: ^11.1.0` and `mikebom:source-manifest: package.json`.
2. **Given** a Cargo workspace declaring `serde = "1.0"` in `Cargo.toml` and resolved to `1.0.203` in `Cargo.lock`, **When** scanned, **Then** exactly one `serde` component appears with PURL `pkg:cargo/serde@1.0.203` and the requirement-range annotation.
3. **Given** the same npm project, **When** scanned once with `--format cyclonedx-json` and once with `--format spdx-2.3-json` and once with `--format spdx-3-json`, **Then** the count of `commander` components is 1 in every format, and the design-tier metadata is present in each format's idiomatic property/annotation channel.
4. **Given** a fixture with 100 declared-and-resolved dependencies (all resolvable in the sibling lockfile), **When** scanned, **Then** the emitted SBOM contains exactly 100 top-level dependency components (down from 200 pre-m191) — a byte-verifiable 50% reduction in duplicate emission.
5. **Given** a scan whose emitted SBOM previously carried the design-tier component's dependency-graph edges (mikebom's `DEPENDS_ON` / `dependsOn` relationships from other components pointing at the design-tier survivor), **When** m191 reconciles design→source, **Then** every incoming edge is rewritten to point at the reconciled source-tier component. No edge is left dangling.

### User Story 2 - Standalone design-tier components with declared-but-not-installed dependencies (Priority: P1)

An operator scans a codebase where a declared dependency has NO matching source-tier resolution — e.g., `optionalDependencies` that failed installation, a freshly-added dep before lockfile refresh, a peer dependency whose provider isn't installed. The design-tier component MUST still appear in the SBOM (its absence is diagnostic — the operator wants to see "declared but not present"). Its PURL MUST be spec-clean: `pkg:npm/commander` (no trailing `@`, no `version` field), NOT the pre-m191 form `pkg:npm/commander@` (trailing `@`, empty `version`).

**Why this priority**: This is the tactical fix for issue #558 (customer scan: 101 design-tier components emitted the malformed trailing-`@` form). Ships together with US1 because the two fixes touch the same emission path, and US2's shape becomes THE canonical shape for US1's "no source-tier resolution" branch. Ships as P1 alongside US1 because the malformed PURL is a customer-visible spec conformance bug affecting vulnerability scanners TODAY, independent of whether reconciliation lands.

**Independent Test**: Scan a fixture npm project declaring `not-yet-installed: "^1.0.0"` in `package.json` but with NO corresponding entry in `package-lock.json`. Assert the emitted CDX 1.6 SBOM contains a component whose PURL is `pkg:npm/not-yet-installed` (no `@`), whose `version` field is omitted from the JSON (not emitted as `""`), whose `bom-ref` is stable and does NOT collide with the source-tier bom-ref shape.

**Acceptance Scenarios**:

1. **Given** an npm project with `optional-dep: "^1.0.0"` declared in `package.json` but no resolution in `package-lock.json`, **When** scanned with `--format cyclonedx-json`, **Then** the emitted component's PURL is `pkg:npm/optional-dep` (no trailing `@`), the `version` field is omitted from the JSON entirely, and the `bom-ref` is stable + non-colliding.
2. **Given** the same fixture, **When** the emitted CDX SBOM is passed through a purl-spec-conformant parser, **Then** the parser accepts the PURL without normalization (no shape drift on round-trip).
3. **Given** the same fixture, **When** scanned with `--format spdx-2.3-json`, **Then** the SPDX `Package` element carries `externalRefs[].referenceLocator = "pkg:npm/optional-dep"` (spec-clean), `versionInfo` is `NOASSERTION` (SPDX convention for unknown version).
4. **Given** the same fixture, **When** scanned with `--format spdx-3-json`, **Then** the SPDX 3 `software_Package` element carries `software_packageUrl = "pkg:npm/optional-dep"` and no `software_packageVersion` property (omitted, not empty).
5. **Given** a mikebom scan whose output is emitted, then re-scanned as a source SBOM (round-trip case), **When** design-tier PURLs are compared before and after the round-trip, **Then** the PURL strings are byte-identical (no normalization-induced drift).

### Edge Cases

- **Manifest declares a range, lockfile resolves to a different name (aliased install)**: e.g., npm alias `"my-alias": "npm:actual-pkg@1.0.0"`. Reconciliation MUST match on the resolved package name, not the alias name. If the design-tier component was keyed on `my-alias` but the source-tier component is `actual-pkg`, collapse into the source-tier `actual-pkg` component and record the alias name as an additional annotation (`mikebom:declared-as`).
- **Manifest declares multiple ranges resolving to the same source-tier component**: e.g., a workspace-level manifest declares `foo: "^1.0"` and a nested workspace manifest declares `foo: "^1.2"`; both resolve to `foo@1.2.3`. The single reconciled source-tier component MUST carry BOTH requirement-range annotations as **multiple `mikebom:requirement-range` property entries**, each paired with a distinguishing `mikebom:source-manifest` property so consumers can reconstruct the range-→-manifest pairing (per FR-004 / Q1). Never drop the multi-declaration information; never collapse ranges into a single JSON-array-encoded string.
- **Manifest declares a range that resolves to multiple source-tier components** (rare, npm workspaces with peer-dep hoisting): e.g., `foo: "^1.0"` resolves to both `foo@1.0.5` (top-level) and `foo@1.2.0` (nested). Reconciliation MUST attach the requirement-range annotation to ALL matching source-tier components, so no consumer downstream loses the "this range covers both" signal.
- **Source-tier component exists WITHOUT a design-tier declaration**: transitive dependency, not directly declared. Emit as-is; no reconciliation happens (there's nothing to reconcile from).
- **Design-tier component with a resolved concrete version already inline** (no range): e.g., `commander: "11.1.0"` (exact-version declaration). Pre-m191 this often still emitted a duplicate design-tier component. Post-m191, if a matching source-tier component exists at the same version, collapse. If not, the design-tier component's PURL carries the version inline (`pkg:npm/commander@11.1.0`) and is spec-clean.
- **Empty range** (unusual but permitted): if the declared range is empty/whitespace-only, treat as no range — emit the design-tier component with a versionless PURL per US2's spec-clean form.
- **Cross-format ID stability**: The reconciled component's PURL, bom-ref (CDX) / SPDXID (SPDX 2.3) / spdxId (SPDX 3) MUST be identical to what the source-tier component would have used pre-m191. This preserves byte-identity for any consumer that indexed on the source-tier component's IDs.
- **Dependency-graph edges pointing at the design-tier component**: In some pre-m191 scans, sibling components (e.g., a parent manifest component) emitted `DEPENDS_ON` edges pointing at the design-tier component. Reconciliation MUST rewrite these edges to point at the reconciled source-tier component so no edge dangles. Edges pointing AT the source-tier component are unchanged.
- **Real-world React Native customer scan** (1998 components, 101 design/source pairs): Post-m191 the same scan MUST produce 1998 − 101 = 1897 components (or fewer if additional reconciliations occur), with zero information loss in the annotation channel.

## Requirements *(mandatory)*

### Functional Requirements

**Design-tier + source-tier reconciliation (US1 — #560)**:

- **FR-001**: System MUST perform a reconciliation pass at emission time (after both design-tier and source-tier component collection are complete, before the format-specific emitter runs) that matches design-tier components to their sibling source-tier components based on (a) canonical PURL name segment, (b) manifest-scope proximity. Proximity semantics per Q2 (Session 2026-07-14): a design-tier declaration in a nested manifest MUST match a source-tier resolution in a workspace-parent lockfile when the workspace claims the nested manifest's directory as a member — reuses the existing m127 root-pick + m149 mainmod demote workspace-parentage plumbing. Independent (non-workspace) directories are NOT walked; the walk stops at the workspace-parent boundary. Standalone (non-workspace) projects match only when the lockfile lives in the same directory as the declaring manifest.
- **FR-002**: When a design-tier component is matched to exactly one source-tier component in the same scope, the design-tier component MUST be removed from the final emitted component list, and its metadata (specifically `mikebom:requirement-range`, `mikebom:source-manifest`, and any dev/prod-scope classification annotations already carried on the design-tier component) MUST be attached to the surviving source-tier component.
- **FR-003**: When a design-tier component is matched to more than one source-tier component (e.g., peer-dep hoisting across workspaces), the reconciliation MUST attach the design-tier metadata to EVERY matching source-tier component. The design-tier component MUST be removed.
- **FR-004**: When multiple design-tier components (from sibling manifests declaring different ranges) reconcile to the same source-tier component, ALL declared ranges MUST be preserved on the surviving source-tier component as **multiple `mikebom:requirement-range` property entries, each paired with a distinguishing `mikebom:source-manifest` property**. The pairing MUST allow a consumer to determine which range came from which manifest by grouping properties on their shared position/index (CDX property array), by grouped-annotation (SPDX 2.3), or by co-emitted graph elements (SPDX 3). NEVER JSON-array-encode multiple ranges into a single property's value.
- **FR-005**: Any dependency-graph edge (CDX `dependencies[].dependsOn`, SPDX 2.3 `DEPENDS_ON` relationship, SPDX 3 `dependsOn` relationship) originating from a sibling component and pointing at the removed design-tier component MUST be rewritten to point at the reconciled source-tier component. No edge MAY dangle after reconciliation.
- **FR-006**: Reconciliation MUST NOT run when NO source-tier match exists — the design-tier component is emitted as a standalone component per US2's requirements. This is the "declared-but-not-installed" branch.
- **FR-007**: Reconciliation MUST NOT introduce new components. Source-tier components that had no corresponding design-tier entry pre-m191 remain emitted as-is (transitive deps, etc.).
- **FR-008**: The reconciled source-tier component's PURL, bom-ref (CDX), SPDXID (SPDX 2.3), and spdxId (SPDX 3) MUST be byte-identical to what the source-tier component would have carried pre-m191. No ID drift.

**Spec-clean versionless PURL for standalone design-tier (US2 — #558)**:

- **FR-009**: When a design-tier component has NO resolved concrete version (declared range but no lockfile resolution, `optionalDependencies` that failed, freshly-added dep before lockfile refresh), System MUST emit the component's PURL WITHOUT a trailing `@` — the canonical purl-spec form is `pkg:<type>/<name>` (or `pkg:<type>/<namespace>/<name>` when a namespace applies) with no `@` and no version segment.
- **FR-010**: In CDX 1.6 output for a versionless design-tier component, the `version` field MUST be omitted from the JSON entirely — NOT emitted as `""` (empty string).
- **FR-011**: In SPDX 2.3 output for a versionless design-tier component, the `versionInfo` field MUST be set to `NOASSERTION` (SPDX 2.3 convention for absent-version).
- **FR-012**: In SPDX 3 output for a versionless design-tier component, the `software_packageVersion` property MUST be omitted from the graph element entirely.
- **FR-013**: The bom-ref (CDX) / SPDXID (SPDX 2.3) / spdxId (SPDX 3) for a versionless design-tier component MUST be the versionless PURL string as-is (e.g., `bom-ref: "pkg:npm/commander"`), matching mikebom's existing "PURL == bom-ref" convention. Per Q3 (Session 2026-07-14). Collision avoidance is guaranteed because source-tier component IDs always carry a non-empty `@version` segment. When a future scan resolves the dep AND reconciliation applies per FR-002, the source-tier's `@version`-carrying ID replaces the versionless design-tier ID — the shift is visible in a diff but the shape stays spec-clean throughout. The ID is stable across scans of the same input because the PURL derivation is deterministic (same manifest name → same PURL segment).
- **FR-014**: Round-tripping a versionless design-tier PURL through a purl-spec-conformant parser MUST produce the same PURL string byte-for-byte (no normalization-induced drift).

**Cross-cutting parity + regression control**:

- **FR-015**: For every ecosystem with existing design-tier + source-tier emission (npm, pip, cargo, maven, gem, pnpm, yarn, composer, ...), the CDX 1.6, SPDX 2.3, and SPDX 3 emissions MUST all reconcile consistently — a component reconciled in one format MUST be reconciled in the other two, and the design-tier annotations MUST appear in each format's idiomatic property channel.
- **FR-016**: Every existing golden fixture that carried duplicate design-tier + source-tier components pre-m191 MUST be regenerated as a documented part of this milestone. Every golden fixture that did NOT carry such duplicates MUST pass byte-identically.
- **FR-017**: The m071 parity catalog rows touching `mikebom:requirement-range` and `mikebom:source-manifest` MUST be updated to reflect the annotation's new residence (now on the source-tier component when reconciliation applies, still on the standalone design-tier component when it doesn't). Directionality of affected rows likely remains `SymmetricEqual` — verify at implementation time.
- **FR-018**: The milestone MUST NOT introduce a new `mikebom:*` annotation. The `mikebom:requirement-range` and `mikebom:source-manifest` annotations already exist (per m127 / m179 / m183 / m184). The `mikebom:declared-as` alias annotation from Edge Cases uses the existing m111 pkg-alias-binding channel. Per CLAUDE.md Principle V (standards-native fields take precedence over `mikebom:` properties).
- **FR-019**: The milestone MUST NOT ship an `--emit-design-when-resolved` (or similar) opt-in-preserve flag. Per issue #560's "Optional flag (deferred)" recommendation: flags for information-removal-vs-preservation calcify into permanent maintenance surface; ship reconciliation-as-default alone and add a flag only if a real consumer asks for the two-component shape.
- **FR-020**: The reconciliation pass MUST emit exactly one INFO-level tracing log per scan summarizing the counts: `reconciled N design-tier components into source-tier siblings; K standalone design-tier components emitted` (per Q4, Session 2026-07-14). MUST additionally emit DEBUG-level per-component logs (subject to the standard `RUST_LOG=debug` filter) recording each reconciliation decision — source component IDs, matched design component IDs, and the annotations transferred. NO document-scope `mikebom:*` annotation is added (per FR-018 + Principle V — reconciliation is a mechanical emit-time transformation, not a diagnostic signal persisted in the SBOM).

### Key Entities *(include if feature involves data)*

- **Design-tier component**: A component whose `mikebom:sbom-tier` is `design` — sourced from a manifest declaration (`package.json`, `pyproject.toml`, `Cargo.toml`, `pom.xml`, `Gemfile`, etc.) but not from a lockfile or a resolved artifact. Carries `mikebom:requirement-range` (the declared version range) + `mikebom:source-manifest` (the path/name of the declaring manifest). Pre-m191: emitted as a standalone component alongside its source-tier counterpart (if any). Post-m191: reconciled into the source-tier counterpart when one exists, or emitted as a versionless standalone component when it doesn't.
- **Source-tier component**: A component sourced from a lockfile / resolved artifact / installed-package DB — carries a concrete version, hashes, licenses, CPE candidates, etc. Referenced today by every downstream security tool as the "authoritative" component. Post-m191: gains the design-tier's requirement-range + source-manifest annotations when reconciliation applies.
- **Reconciliation pair**: A `(design-tier component, source-tier component[s])` tuple matched at emission time by (canonical PURL name segment, manifest-scope proximity). Ephemeral — lives only during the reconciliation pass; not persisted.
- **Versionless PURL**: A purl-spec-canonical PURL string without an `@version` segment, of the form `pkg:<type>/<name>` or `pkg:<type>/<namespace>/<name>`. Used for standalone design-tier components with no resolved version.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For every fixture with a design-tier component that has a matching source-tier resolution in the same scope, the emitted SBOM contains exactly ONE component representing that dependency (down from two pre-m191). Measured as `unique(name) - unique_pre_m191(name) == +1 dedup ratio` per reconciliation pair for regression fixtures.
- **SC-002**: For every reconciled component, 100% of the pre-m191 design-tier metadata (specifically `mikebom:requirement-range` and `mikebom:source-manifest`) is present on the surviving source-tier component. Measured as: `count(reconciled_components_with_requirement_range) == count(pre_m191_design_tier_components_with_requirement_range)` on any test fixture.
- **SC-003**: For every standalone design-tier component (no source-tier match), the emitted PURL contains no `@` character and the format-specific version field is omitted / `NOASSERTION` per format convention. Measured as: `grep -oE '"purl":\s*"pkg:[^@"]*@[^"?]*"' | grep -c '"' == 0` for standalone design-tier components in the output.
- **SC-004**: Round-tripping every versionless design-tier PURL through a purl-spec parser produces the same string byte-for-byte. Measured via a fuzz-style test that parses + reserializes 100 synthetic versionless PURLs and asserts identity.
- **SC-005**: The customer React Native scan reference (or the largest ecosystem fixture in the test corpus with design-tier + source-tier pairs) produces at least a 5% reduction in total component count when reconciliation applies (based on the 101/1998 ratio ≈ 5% reported on the customer scan).
- **SC-006**: Every existing golden fixture without design-tier + source-tier pairs (or where all such pairs were already reconciled correctly by pre-m191 logic) produces byte-identical emission before and after m191 — the byte-identity safety net for the milestone.
- **SC-007**: For every affected format (CDX 1.6, SPDX 2.3, SPDX 3), a schema validator accepts the reconciled emission without errors — no new schema violations introduced.

## Assumptions

- The existing reader plumbing (m127 root-pick, m149 mainmod demote, m179 lifecycle-scope, m183 pip-extras optional-dep, m184 maven-gradle optional-dep) already correctly attaches `mikebom:sbom-tier` (`design` or `source`) and `mikebom:source-manifest` to every component. This milestone reads those existing annotations to perform the reconciliation match — it does not introduce a new tier-classification step.
- Design-tier + source-tier pairs today carry the same PURL name segment (case-sensitive) — reconciliation matches on that segment plus scope. If a reader ever emits a design-tier with a different name than its resolved source-tier counterpart (rare — usually only when npm aliasing is at play), the Edge Cases branch covers it via the `mikebom:declared-as` annotation channel.
- "Manifest-scope proximity" is determined by shared origin — same manifest file for the declaration + a lockfile in the same directory (or in a workspace-parent directory that resolves the declaration). The existing per-reader plumbing already tracks this via `mikebom:source-manifest` values; the reconciliation pass consumes them.
- No new Cargo dependencies are required. The reconciliation pass is a pure in-memory transformation over the assembled component list — reuses existing `serde_json`, workspace types (`ResolvedComponent`, `Purl`), and format emitters. Consistent with every recent milestone since m185 and this repo's zero-new-deps posture.
- The `mikebom:sbom-tier` property is a mikebom-specific field (per m127/m183) but pre-dates m191; the Native-first audit (Principle V) applied at those milestones stands unchanged.
- The customer React Native monorepo scan reference from #558 / #560 (1998 components, 101 design/source pairs) is treated as an aspirational validation target. Because the actual customer artifact is not check-in-able, this milestone ships with a synthetic multi-package npm fixture that reproduces the shape and asserts the 5% reduction ratio (matching the customer report).
- The milestone-071 parity extractor infrastructure already covers `mikebom:requirement-range` (added in m183/m184) and `mikebom:source-manifest` (added in m127). Their Directionality is likely already `SymmetricEqual`. Verify at plan time; if any extractor is single-direction, adjust as part of this milestone.
- Ecosystems in scope for immediate coverage: npm (package.json + lockfile), pnpm (pnpm-lock), yarn (yarn.lock), pip (pyproject.toml + poetry.lock / uv.lock / requirements.txt), cargo (Cargo.toml + Cargo.lock), maven (pom.xml), gem (Gemfile + Gemfile.lock). Composer (PHP), gradle (Kotlin/Groovy DSL), sbt (Scala), rebar (Erlang), mix (Elixir) MAY be covered transitively if the design-tier reader already produces components for them; otherwise those ecosystems are follow-up work.
- The two fixes ship as a single milestone (m191) at the user's explicit direction (2026-07-14 triage response). Non-goals for m191: an opt-in preservation flag (per #560's own recommendation), cross-manifest reconciliation across independent workspaces (out of scope), any change to the design-tier reader plumbing (this is purely an emission-time pass).
