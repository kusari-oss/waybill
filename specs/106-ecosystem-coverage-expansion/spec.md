# Feature Specification: Ecosystem Coverage Expansion (Phase 1)

**Feature Branch**: `106-ecosystem-coverage-expansion`
**Created**: 2026-05-31
**Status**: Draft
**Input**: User description: "Expand mikebom filesystem package-detection coverage with four new ecosystem parsers: NuGet, uv, Gradle, and bun. Closes issues #275, #276, #277, #278."

## Context

mikebom's filesystem package-detection layer (`scan_fs/package_db/`) covers a strong set of mainstream ecosystems today — Cargo, npm, pip (`requirements.txt`, `pipfile.lock`, `poetry.lock`), Maven (`pom.xml` + JAR `META-INF`), Gem, dpkg, apk, rpm, plus the C/C++ readers shipped through milestone 105. Four meaningful gaps remain for projects scanned at the source-tree level:

| Issue | Ecosystem | Today's behavior | Coverage gap |
|---|---|---|---|
| [#275](https://github.com/kusari-sandbox/mikebom/issues/275) | **.NET / NuGet** | Zero components emitted for any .NET source tree | No `.csproj` / `.vbproj` / `.fsproj` parser; no `packages.lock.json` parser. PURL + deps.dev enrichment are already wired downstream. |
| [#276](https://github.com/kusari-sandbox/mikebom/issues/276) | **uv (modern Python)** | Zero components when only `uv.lock` is present | No `uv.lock` reader; PyPI infrastructure already wired. |
| [#277](https://github.com/kusari-sandbox/mikebom/issues/277) | **Gradle (JVM)** | Empty Java SBOMs from Gradle source trees pre-build | No `gradle.lockfile` / `buildscript-gradle.lockfile` parser; Maven `pkg:maven/...` PURL emission and deps.dev enrichment already wired. |
| [#278](https://github.com/kusari-sandbox/mikebom/issues/278) | **Bun (JS)** | Zero components from Bun-using projects | No `bun.lock` parser; npm `pkg:npm/...` PURL emission already wired. |

This milestone closes the four gaps in priority order. All four new parsers slot into the existing `scan_fs/package_db/<ecosystem>/` dispatch architecture and produce PURLs in already-wired ecosystems (`pkg:nuget/`, `pkg:pypi/`, `pkg:maven/`, `pkg:npm/`) so downstream resolution + enrichment work without changes. Each new parser has Apache-2.0 prior art in [osv-scalibr v0.4.5](https://github.com/google/osv-scalibr) (the Go-language equivalent), which serves as a reference for edge-case handling but does not introduce a copy-paste obligation — the implementations are clean-room Rust.

## Clarifications

### Session 2026-05-31

- Q: For uv and Bun workspaces (monorepo projects with multiple members), how should mikebom emit workspace members? → A: **Emit workspace members as first-class components with intra-workspace dependency edges preserved, plus a synthetic virtual root representing the monorepo.** Specifically: (1) each workspace member emits as a component with the ecosystem-native PURL (`pkg:pypi/<name>@<version>` for uv, `pkg:npm/<name>@<version>` for Bun) and `mikebom:component-role: "main-module"` (reuses the existing annotation introduced by milestone 064 cargo / 066 npm / 068 pip); (2) the monorepo root emits as a synthetic virtual component derived from the root manifest's `name` (or `"workspace-root"` placeholder when absent) with `mikebom:component-role: "workspace-root"`; (3) the virtual root has `dependsOn` edges to each workspace member; (4) intra-workspace dependency edges between members are preserved (e.g., a Bun workspace `apps/web` declaring `"@my/shared": "workspace:*"` produces a `dependsOn` edge from `web` to `shared`); (5) independent workspace members produce NO edge between them — the dependency graph reflects only ACTUAL declared edges, not co-location. The user's worked example: mikebom-cli and mikebom-common are both Cargo workspace members; mikebom-cli's `Cargo.toml` declares `mikebom-common = { path = "..." }`, so the SBOM has `mikebom-cli → mikebom-common` but no spurious edges to workspace siblings that mikebom-cli doesn't import.
- Q: Should the NuGet reader support Central Package Management (CPM) via `Directory.Packages.props`? → A: **Yes — read `Directory.Packages.props` at the repo root (and via walk-up search), resolve `.csproj` `<PackageReference Include="X"/>` entries against `<PackageVersion Include="X" Version="..."/>` in the props file.** Cover both legacy `Version=`-on-the-`PackageReference` form AND the modern CPM form in a single milestone — Visual Studio 2022+ and `dotnet 8+` project templates increasingly default to CPM, so without it, scanning current Microsoft reference templates produces unversioned components. Implementation cost is one additional XML parser file (`nuget/directory_packages_props.rs`) plus a simple Include-to-PackageVersion lookup in the .csproj resolver. The lookup fires unconditionally — the `<ManagePackageVersionsCentrally>true</ManagePackageVersionsCentrally>` flag is treated as informational only; a `Directory.Packages.props` file present in the walk-up chain is sufficient signal that CPM is in use. Mixed projects (some refs with `Version=`, some without) resolve each ref independently — explicit `Version=` wins, otherwise fall back to the props lookup.
- Q: For Gradle `buildscript-gradle.lockfile` entries, should mikebom emit the native CDX `scope: "excluded"` field in addition to the `mikebom:lifecycle-scope: "build"` annotation? → A: **Yes — emit both.** Matches the milestone-052 lifecycle-scope-to-CDX-scope mapping pattern that the existing Maven, npm, and Cargo readers already use for `build`/`dev`/`test`-scoped dependencies. Downstream vuln scanners that honor CDX `scope` filtering get a clean "this is a build plugin, not deployment-tier" signal automatically; operators who want build-plugin CVEs surfaced explicitly can re-include them with `--exclude-scope ""` or pipeline filtering. Skipping the native scope field would create an inconsistency with the rest of mikebom's lifecycle-scope handling. NO change to the SPDX 2.3 / SPDX 3 side — those emit the `mikebom:lifecycle-scope` property only (the existing milestone-052 SPDX behavior).
- Q: Real-world `bun.lock` files contain `// comments` (JSONC syntax) which `serde_json` rejects. How should mikebom handle this? → A: **Pre-process the file by stripping `//` line comments and `/* */` block comments before parsing with `serde_json::from_str`.** No new Cargo dependency (`serde_jsonc` / `json5` not in the workspace closure). Strip logic is ~20 LOC of straightforward Rust well-tested against the JSONC spec. Implementation pattern mirrors existing comment-stripping helpers in the codebase: `gem::strip_ruby_comment` at `scan_fs/package_db/gem.rs:149` and `golang::legacy::strip_line_comment` at `scan_fs/package_db/golang/legacy.rs:82-84`. Rejecting commented `bun.lock` files would be hostile — every real Bun 1.2+ lockfile has at least the top-of-file `// bun: lockfileVersion: 1` version-marker comment.
- Q: Should NuGet `<PackageReference>` entries with `PrivateAssets="All"` (or equivalent `IncludeAssets` patterns that exclude `runtime`) be tagged as build-only via `mikebom:lifecycle-scope: "build"` + CDX `scope: "excluded"`? → A: **Yes — apply the same lifecycle-scope mapping as Gradle's `buildscript-gradle.lockfile` (Q3).** Cross-ecosystem consistency: `PrivateAssets="All"` is the .NET analog of a build-only dependency (canonical examples: `Microsoft.SourceLink.GitHub`, `Microsoft.CodeAnalysis.NetAnalyzers`, source generators). Implementation cost is one extra XML attribute check in the .csproj parser. Mapping rules: (1) `PrivateAssets="All"` → `mikebom:lifecycle-scope: "build"` + CDX `scope: "excluded"`; (2) `IncludeAssets="..."` patterns that omit `runtime` (e.g., `IncludeAssets="build,buildMultitargeting"`) → same treatment; (3) `PrivateAssets="None"` or absent → runtime default (no lifecycle-scope annotation, no CDX scope). The `ExcludeAssets="runtime"` form is also recognized as build-only. SPDX side emits the annotation only (no native SPDX scope field exists).

The five-issue scope is one coordinated push rather than four separate milestones because:

- All four parsers fit the **same shape of work** (regex / TOML / XML / line-format decoder → existing `PackageDbEntry` shape → existing dispatcher).
- All four reuse **existing PURL types** and downstream pipeline plumbing — no new resolver, no new enrichment, no new parity-catalog rows.
- All four can ship **independently** (each user story corresponds to one sub-PR), but coordinating them under one spec means the operator-facing docs (`docs/ecosystems.md`, the coverage matrix) get a single coherent update instead of four tiny ones.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Modern Python project using uv (Priority: P1)

A developer's Python project uses [uv](https://docs.astral.sh/uv/) (Astral) for dependency management. The project has a `pyproject.toml` and a `uv.lock` at the repo root, but no `requirements.txt`, no `pipfile.lock`, no `poetry.lock`. The developer runs `mikebom sbom scan --path .` and expects every locked package from `uv.lock` to appear as an SBOM component with the version uv resolved.

**Why this priority**: uv is the fastest-growing modern Python package manager (Astral hit 100M+ monthly downloads in 2025), and mikebom's coverage of "modern Python" is otherwise complete (poetry / pipfile / requirements.txt). Adding uv.lock is the smallest possible per-LOC unlock for the largest currently-blocked user segment. Implementation is tiny — uv.lock is TOML with a `[[package]]` array structurally identical to `Cargo.lock`, so the existing `cargo.rs` TOML helpers may be directly reusable.

**Independent Test**: Add a fixture `mikebom-cli/tests/fixtures/golden_inputs/uv_lock/` containing a `uv.lock` declaring 3–5 packages (e.g., `httpx 0.27.2`, `pydantic 2.9.2`). Scan the fixture; assert all packages emerge as `pkg:pypi/<name>@<version>` with their dependency edges populated from the `[[package.dependencies]]` arrays.

**Acceptance Scenarios**:

1. **Given** a project with `uv.lock` declaring `httpx 0.27.2`, **When** scanned, **Then** an SBOM component `pkg:pypi/httpx@0.27.2` emerges with `mikebom:source-files: ".../uv.lock"`.
2. **Given** a `uv.lock` with `[[package.dependencies]]` arrays, **When** scanned, **Then** the resulting SBOM has `dependsOn` edges (CDX) / `DEPENDS_ON` relationships (SPDX) between the resolved packages matching the lockfile's dependency graph.
3. **Given** a project with both `uv.lock` AND `poetry.lock`, **When** scanned, **Then** mikebom emits each unique PURL exactly once (deduplication via existing PURL key) and does NOT double-count. The lifecycle-scope tagging matches whichever lockfile resolved first per existing pip-dispatcher precedence.
4. **Given** a `uv.lock` with version-constraint-only entries (no `version =` field at top level — uv supports source-only entries), **When** scanned, **Then** mikebom emits a `tracing::warn!` naming the unresolvable entry and continues (FR-013-equivalent warn-and-continue).
5. **Given** a uv workspace (`[tool.uv.workspace]` in the root `pyproject.toml` listing members under `apps/web`, `apps/api`, and `libs/shared`, where `apps/web` declares a workspace-source dependency on `libs/shared`), **When** scanned, **Then** mikebom emits: (a) a synthetic workspace-root component derived from the root `pyproject.toml`'s `name` with `mikebom:component-role: "workspace-root"`, (b) one component per workspace member as `pkg:pypi/<name>@<version>` with `mikebom:component-role: "main-module"`, (c) `dependsOn` edges from the workspace-root to each member, (d) a `dependsOn` edge from `apps/web`'s component to `libs/shared`'s component (the declared intra-workspace dep), (e) NO edge between `apps/web` and `apps/api` (no declared dep), (f) external PyPI deps appearing as transitives with edges from the appropriate member.

### User Story 2 — JavaScript project using Bun (Priority: P1)

A developer's JS/TS project uses [Bun](https://bun.sh/) as the package manager and runtime. The repo has a `bun.lock` (the modern Bun 1.2+ text format, not the older binary `bun.lockb`), plus a `package.json`. The developer runs `mikebom sbom scan --path .` expecting every declared dependency from `bun.lock` to appear.

**Why this priority**: Bun is a fast-growing JS toolchain; mikebom's npm/pnpm coverage is otherwise complete. Implementation parallels the existing `package_lock.rs` and `pnpm_lock.rs` patterns — one file in the existing `npm/` module. The older binary `bun.lockb` format is explicitly out of scope per the issue.

**Independent Test**: Add a fixture `mikebom-cli/tests/fixtures/golden_inputs/bun_lock/` with a `bun.lock` declaring multiple packages. Scan; assert `pkg:npm/...` components emerge with correct versions and `dependencies[]` edges.

**Acceptance Scenarios**:

1. **Given** a project with `bun.lock` declaring `lodash@4.17.21`, **When** scanned, **Then** `pkg:npm/lodash@4.17.21` emerges with `mikebom:source-files: ".../bun.lock"`.
2. **Given** a `bun.lock` declaring scoped packages (`@types/node@22.5.0`), **When** scanned, **Then** the PURL encodes the scope correctly as `pkg:npm/%40types/node@22.5.0`.
3. **Given** a project with both `bun.lock` AND `package-lock.json`, **When** scanned, **Then** components are deduplicated by canonical PURL; the npm dispatcher's existing precedence (lockfile > node_modules > root package.json) decides which reader "owns" the source-files annotation.
4. **Given** a `bun.lock` containing a workspace declaration (Bun monorepo pattern with `workspaces: ["packages/*"]` in the root `package.json`), **When** scanned, **Then** mikebom follows the workspace-root + member emission pattern from the Clarifications: (a) a synthetic workspace-root component named from the root `package.json`'s `name` with `mikebom:component-role: "workspace-root"`, (b) one component per workspace member as `pkg:npm/<name>@<version>` with `mikebom:component-role: "main-module"`, (c) `dependsOn` edges from workspace-root to each member, (d) intra-workspace edges preserved where a member declares `"@my/shared": "workspace:*"` (the Bun workspace-source convention), (e) external npm deps as transitives.

### User Story 3 — JVM project using Gradle source-tree scan (Priority: P2)

A developer's Java/Kotlin/Scala project uses Gradle with [dependency locking](https://docs.gradle.org/current/userguide/dependency_locking.html) enabled. The developer runs `mikebom sbom scan --path .` against the source tree (pre-build, no JARs on disk) expecting every locked Gradle dependency to appear with `pkg:maven/...` PURLs that match what the existing Maven `pom.xml` reader would have produced for an equivalent Maven project.

**Why this priority**: Gradle is one of the two dominant JVM build tools; mikebom's pre-build Java coverage today is Maven-only. Inspector PR scans run on source trees, not built artifacts, so any Gradle repo currently produces empty Java SBOMs. Implementation effort is medium because the lockfile format is well-documented + line-oriented (no XML), but the runtime-vs-buildscript distinction adds lifecycle-scope tagging work.

**Acceptance Scenarios**:

4. **Given** a project with both `gradle.lockfile` AND a `pom.xml` (rare hybrid), **When** scanned, **Then** components from both readers are deduplicated by PURL; the existing Maven dispatcher precedence decides the source-files annotation.
5. **Given** an empty-configurations line (`group:name:version=` with no configs after `=`), **When** scanned, **Then** the component is still emitted with default lifecycle scope and a `tracing::warn!` event noting the empty configs.

### Edge Cases

- **uv.lock — workspace members**: a `uv.lock` from a workspace (multiple members in `pyproject.toml`'s `[tool.uv.workspace]`) lists each member as a `[[package]]` entry with `source = { workspace = true }`. Per the Clarifications session: emit workspace members as first-class `pkg:pypi/<name>@<version>` components with `mikebom:component-role: "main-module"`, plus a synthetic workspace-root component above them. Intra-workspace edges between members are preserved; independent members get no spurious edges. The `mikebom:source-mechanism` annotation is NOT used here (this is a PyPI-ecosystem component, not a new closed-enum source-mechanism value).
- **uv.lock — git/path sources**: a `[[package]]` with `source = { git = "https://..." }` or `source = { path = "../foo" }` emits as `pkg:git+https://<url>@<rev>` or `pkg:generic/<name>` (mirrors the milestone-105 idf-component-local pattern for path deps).
- **bun.lock — overrides**: Bun's `bun.lock` can declare overrides (`"overrides": { "lodash": "4.17.21" }`). The override version wins; mikebom does NOT also emit the un-overridden version as a separate component.
- **bun.lock — JSONC comments**: real-world `bun.lock` files contain at least a top-of-file `// bun: lockfileVersion: 1` marker plus optional inline comments. Per FR-003, mikebom strips both `//` line comments and `/* */` block comments before parsing as JSON. Comments INSIDE string values are preserved (the strip is comment-aware about string boundaries).
- **Gradle — invalid version format**: a Gradle lockfile line with a malformed version string (e.g., `group:name:1.0+nightly`) is parsed as best-effort and emitted with a `tracing::warn!` if the resulting PURL fails the `Purl::new` validation.
- **Gradle — group with dots**: standard Gradle deps have multi-segment group names (`com.google.guava`). The `pkg:maven/<group>/<name>@<version>` PURL uses the full dotted group as the namespace (matches existing Maven PURL conventions).
- **NuGet — case-insensitivity**: package names in NuGet are case-insensitive on the registry but mikebom preserves the case from the source file. The dedup pipeline (FR-015 from milestone 105) treats different cases as the same canonical PURL via the existing PURL normalization.
- **NuGet — multiple project files**: a typical .NET solution has multiple `.csproj` files. mikebom unions the dependencies; duplicates are collapsed by canonical PURL with `mikebom:source-files` listing all source files.
- **NuGet — `<TargetFramework>` differences**: a `<PackageReference>` may be conditional on the target framework (`<PackageReference Include="..." Version="..." Condition="'$(TargetFramework)' == 'net8.0'"/>`). mikebom emits the package regardless; condition expression is recorded as a `mikebom:lifecycle-scope-guard` annotation but NOT evaluated.
- **NuGet — Central Package Management lookup miss**: when a `<PackageReference>` has no `Version=` AND no matching `<PackageVersion>` in any walk-up `Directory.Packages.props`, mikebom emits `pkg:nuget/<name>@unresolved` plus a `tracing::warn!` event. See FR-007 / FR-007a + US4 scenario 7.
- **NuGet — `Directory.Packages.props.user` overrides**: MSBuild also honors a `Directory.Packages.props.user` file as a local override. mikebom skips this file (it's typically gitignored and not part of the repo's authoritative state); a `tracing::info!` event records the skip.
- **NuGet — `Directory.Build.props` inheritance**: some repos embed `<PackageVersion>` entries in `Directory.Build.props` rather than `Directory.Packages.props`. mikebom does NOT walk `Directory.Build.props` in this milestone; if real-world demand surfaces, follow-up work can add it.
- **NuGet — `PrivateAssets`/`IncludeAssets`/`ExcludeAssets` build-only deps**: per Clarifications Q5 + FR-007b, `<PackageReference>` entries with `PrivateAssets="All"` (or equivalent `IncludeAssets`/`ExcludeAssets` patterns that omit `runtime`) are tagged build-only. Canonical .NET examples: `Microsoft.SourceLink.GitHub`, `Microsoft.CodeAnalysis.NetAnalyzers`, source generators. CDX output gets native `scope: "excluded"`; SPDX gets the `mikebom:lifecycle-scope: "build"` annotation only.
- **Polyglot scan robustness**: per milestone-105 SC-008, encountering an unsupported file shape in any one reader MUST NOT abort the whole scan. The new readers MUST also warn-and-skip on parse failures rather than fail closed.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: When mikebom encounters a `uv.lock` file anywhere in the scan tree, it MUST parse it as TOML with a `[[package]]` array and emit one SBOM component per resolved entry with PURL `pkg:pypi/<name>@<version>` and annotation `mikebom:source-files: "<path/to/uv.lock>"`.
- **FR-002**: For each `uv.lock` `[[package]]` entry that declares `[[package.dependencies]]` sub-arrays, mikebom MUST emit `dependsOn` (CDX) / `DEPENDS_ON` (SPDX) relationships between the resolved component and each named dependency that also resolves to a `[[package]]` entry in the same lockfile.
- **FR-003**: When mikebom encounters a `bun.lock` file at the scan root, it MUST parse it (JSONC — JSON with Comments — per Bun 1.2+) and emit one SBOM component per declared dependency with PURL `pkg:npm/<name>@<version>` (scoped packages emit as `pkg:npm/%40<scope>/<name>@<version>`). The parser MUST pre-process the file to strip `//` line comments and `/* */` block comments before handing the result to `serde_json::from_str` — every real-world `bun.lock` has at least the top-of-file `// bun: lockfileVersion: 1` marker. No new Cargo dependency; the strip helper is implemented from scratch in-tree (pattern mirrors `gem::strip_ruby_comment` and `golang::legacy::strip_line_comment` — single-line and block-comment scanners that respect string-literal boundaries). The older binary `bun.lockb` format is out of scope.
- **FR-004**: For each `bun.lock` dependency with a `dependencies` map, mikebom MUST emit relationship edges between the resolved component and each named dependency.
- **FR-005**: When mikebom encounters a `gradle.lockfile` or `buildscript-gradle.lockfile` anywhere in the scan tree, it MUST parse the line format `group:name:version=<configs>` and emit one SBOM component per entry with PURL `pkg:maven/<group>/<name>@<version>` and annotation `mikebom:source-files: "<path>"`. The `# Generated by Gradle...` header line MUST be skipped without error.
- **FR-006**: Components from `buildscript-gradle.lockfile` MUST be tagged `mikebom:lifecycle-scope: "build"` AND carry the native CDX `scope: "excluded"` field (matching the milestone-052 lifecycle-scope-to-CDX-scope mapping that existing Maven, npm, and Cargo readers use for build/dev/test-scoped deps). The SPDX 2.3 / SPDX 3 side emits only the `mikebom:lifecycle-scope` annotation — no native SPDX scope field exists (also consistent with milestone-052). Components from `gradle.lockfile` default to the runtime lifecycle scope (no `mikebom:lifecycle-scope` annotation, no CDX `scope` field — matching the existing Maven runtime-default convention).
- **FR-007**: When mikebom encounters a `*.csproj`, `*.vbproj`, or `*.fsproj` file anywhere in the scan tree, it MUST parse the XML and extract every `<PackageReference Include="..."/>` element. Each extracted entry emits as `pkg:nuget/<name>@<version>`. Version resolution: (a) when the element has a `Version="..."` attribute, that wins (legacy form); (b) otherwise, look up the `<PackageVersion Include="X" Version="..."/>` entry in the nearest `Directory.Packages.props` (Central Package Management form — see FR-007a); (c) if neither resolves, emit with `version: "unresolved"` and a `tracing::warn!` event naming the .csproj file and the missing package.
- **FR-007a**: When mikebom encounters a `Directory.Packages.props` file, it MUST parse the XML and extract every `<PackageVersion Include="..." Version="..."/>` element into a per-directory lookup table. The `.csproj` resolver from FR-007 walks UP the directory tree from each project file to find the nearest `Directory.Packages.props` (mirrors MSBuild's actual lookup behavior). The `<ManagePackageVersionsCentrally>true</ManagePackageVersionsCentrally>` property element is treated as informational only — presence of a `Directory.Packages.props` file is sufficient signal that CPM is in use.
- **FR-007b**: For each `<PackageReference>` element, mikebom MUST inspect the `PrivateAssets`, `IncludeAssets`, and `ExcludeAssets` attributes to determine lifecycle scope. (1) `PrivateAssets="All"` → tag the component with `mikebom:lifecycle-scope: "build"` + native CDX `scope: "excluded"`; (2) `IncludeAssets="..."` patterns that omit `runtime` (e.g., `IncludeAssets="build,buildMultitargeting"`) → same treatment; (3) `ExcludeAssets="runtime"` → same treatment; (4) absent / `PrivateAssets="None"` → runtime default (no annotation, no CDX scope field). Attribute parsing is case-insensitive (MSBuild treats them so) but value matching uses canonical lowercase. SPDX 2.3 / SPDX 3 emit the `mikebom:lifecycle-scope` annotation only (no native SPDX scope field).
- **FR-008**: When mikebom encounters a `packages.lock.json` file adjacent to a project file, it MUST parse the JSON and use it as the authoritative source for resolved versions, INCLUDING transitive dependencies. The project file is then only used for the direct-dependency set; lockfile entries provide the version pins.
- **FR-009**: All new readers MUST emit `mikebom:source-files` annotations naming the manifest file that contributed the entry. When multiple files resolve the same canonical PURL (e.g., `.csproj` + `packages.lock.json`), `mikebom:source-files` MUST list both paths (comma-joined, matching the existing convention).
- **FR-010**: All new readers MUST be polyglot-safe: a parse error in any one of them MUST emit a structured `tracing::warn!` event naming the offending file and continue scanning. Adjacent ecosystem readers MUST be unaffected (matches milestone-105 SC-008).
- **FR-011**: All new readers MUST integrate with the existing dispatcher in `scan_fs/package_db/mod.rs::read_all` such that their components participate in the milestone-105 dedup pipeline (FR-015) when applicable. Practically: emit `PackageDbEntry` values that flow through the existing pipeline; no new dispatcher-level wiring beyond adding the `<reader>::read(...)` call.
- **FR-012**: All new readers MUST work in `--offline` mode. Version resolution is entirely local-file driven — no network calls to NuGet, PyPI, Maven Central, or npm registries during a scan.
- **FR-013**: All new ecosystems MUST be added to `docs/ecosystems.md`'s coverage matrix. For each: ecosystem name, the file shapes recognized, the PURL type emitted, and a one-line note about transitive-dependency support.
- **FR-014**: The existing `--exclude-scope` CLI flag MUST work consistently for the new readers. Specifically: `--exclude-scope build,dev,test` filters out `buildscript-gradle.lockfile` components (Gradle build classpath) and is a no-op for the other three ecosystems (uv/bun/NuGet don't currently distinguish build-scope at the lockfile level).
- **FR-015**: When a uv or Bun workspace is detected (root `pyproject.toml` declares `[tool.uv.workspace]` with `members`; root `package.json` declares `workspaces: [...]`), mikebom MUST emit: (a) a synthetic workspace-root component with `mikebom:component-role: "workspace-root"` whose name derives from the root manifest's declared `name` (fallback `"workspace-root"` when the field is absent or empty); (b) one component per workspace member with `mikebom:component-role: "main-module"` and the ecosystem-native PURL form (`pkg:pypi/<name>@<version>` for uv, `pkg:npm/<name>@<version>` for Bun); (c) `dependsOn` (CDX) / `DEPENDS_ON` (SPDX) edges from the workspace-root to each member component; (d) intra-workspace dependency edges between members ONLY when the source manifest declares them (e.g., a Bun member declaring `"@my/shared": "workspace:*"` or a uv member declaring a workspace-source dependency); independent members MUST NOT have spurious edges between them.

### Key Entities

- **uv lockfile** — a TOML file at `<project>/uv.lock` containing a top-level `[[package]]` array. Each entry has `name`, `version`, optional `source` (path/git/registry), and optional `[[package.dependencies]]` sub-array. Mirrors `Cargo.lock`'s shape structurally.
- **bun lockfile** — a text file at `<project>/bun.lock` containing per-package entries with name, version, and optional `dependencies` map. Distinct from the older binary `bun.lockb` (out of scope).
- **Gradle lockfile** — a text file at `<project>/gradle.lockfile` (runtime classpath) or `<project>/buildscript-gradle.lockfile` (build-script classpath). Each line: `group:name:version=<comma-separated-configs>`. Preceded by a `# Generated by Gradle...` header.
- **NuGet project file** — an XML file at `<project>/<name>.{csproj,vbproj,fsproj}` containing `<PackageReference Include="..." Version="..."/>` elements inside `<ItemGroup>` blocks.
- **NuGet lockfile** — a JSON file at `<project>/packages.lock.json` containing per-target-framework resolved dependency trees. When present, it is the authoritative source for versions; the project file's `<PackageReference>` provides only the direct-dependency set.
- **NuGet Central Package Management (CPM) props file** — an XML file at `<repo-root>/Directory.Packages.props` (or any ancestor of a `.csproj` file via MSBuild's walk-up resolution) containing `<PackageVersion Include="..." Version="..."/>` entries inside `<ItemGroup>`. When CPM is in use, `.csproj` files have `<PackageReference Include="..."/>` without a `Version=` attribute and rely on the props-file lookup for resolution. See FR-007a.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001** (uv coverage): On a benchmark suite of 5 representative uv-using open-source projects (e.g., projects that use `uv lock` for reproducible builds), mikebom emits 100% of the dependencies declared in `uv.lock` as SBOM components with matching versions. Verified against `uv pip list` output as ground truth.
- **SC-002** (bun coverage): On a benchmark of 5 open-source Bun-using projects, mikebom emits 100% of the `bun.lock` resolved dependencies with matching versions.
- **SC-003** (Gradle coverage): Scanning a Gradle project with dependency locking enabled (e.g., Spring Boot's `gradle.lockfile` from a canonical samples repo) emits at least one component per locked entry. Empty-Java-SBOM regression for Gradle source trees is closed.
- **SC-004** (NuGet coverage): Scanning a sample .NET solution with `.csproj` files emits at least one component per `<PackageReference>`. A solution with `packages.lock.json` present additionally surfaces transitive dependencies absent from the `.csproj` files.
- **SC-005** (Parity preservation): All 33 existing golden fixtures continue to pass byte-identity after this milestone. The four new ecosystems add new golden fixtures (one per reader, CDX + SPDX 2.3 + SPDX 3.0.1) but do not modify existing ones.
- **SC-006** (Polyglot robustness): A scan of a tree containing manifests for all four new ecosystems plus an unsupported variant (e.g., `bun.lockb` binary format) emits all four ecosystems' components and a warn message for the unsupported variant — no scan abort.

## Assumptions

- **No new Cargo dependencies**: All four parsers use crates already in the workspace dependency closure. TOML parsing reuses the existing `toml = "0.8"` dep (used by `cargo.rs`). XML parsing reuses `quick-xml = "0.31"` (used by `maven.rs`). JSON uses `serde_json` (pervasive). Line-oriented parsing for Gradle uses `std::str::Lines`.
- **Reference-only use of osv-scalibr**: the existing Go parsers in scalibr v0.4.5 inform edge-case handling (header detection, malformed-line skip, empty-configs handling) but mikebom's Rust implementations are clean-room. No code copied; no Apache-2.0 LICENSE file additions needed.
- **Reuse of milestone-105 infrastructure**: the cross-reader dedup pipeline (`scan_fs/dedup.rs`) and parity catalog rows (C55 / C56 / C57) are reused as-is. No new dedup precedence tier needed — the new ecosystems all fall into the "manifest-mode" tier alongside Conan, vcpkg, etc.
- **Zero new C/C++ scope**: this milestone is parallel to milestone 105's C/C++ work; no overlap. The milestone-105 readers (CPM.cmake, conanfile.py, west.yml, idf_component.yml, vcpkg-classic, git-submodule) ship independently.
- **PURL ecosystem types are stable**: `pkg:nuget/`, `pkg:pypi/`, `pkg:maven/`, `pkg:npm/` are all in the package-url spec today (no speculative ecosystem namespaces like milestone-105's `pkg:idf/`).
- **Source-tree scan model**: all four readers operate on source trees / lockfiles, NOT on built artifacts. JAR / .nupkg / .whl / .tgz inspection is out of scope and is covered by separate existing readers (`maven.rs` for JARs, `pip/dist_info.rs` for installed wheels, etc.).
- **Polyglot dedup**: in projects that have multiple lockfiles from different managers (e.g., a Python repo with both `uv.lock` AND `poetry.lock`), the dedup pipeline collapses by canonical PURL. Lifecycle-scope tagging follows the first reader to resolve each PURL per the existing pip dispatcher precedence.
- **Workspace handling**: resolved during the Clarifications session — workspace members emit as first-class `pkg:pypi/...` / `pkg:npm/...` components with `mikebom:component-role: "main-module"`. A synthetic workspace-root component sits above them with `mikebom:component-role: "workspace-root"`. Intra-workspace dependency edges are preserved per declared sources only; independent members get no edges between them. See FR-015 and Clarifications Q1 for the full policy. This is a deliberate inversion of the spec's original draft "workspace meta-package NOT emitted" stance — the user's worked example (mikebom-cli + mikebom-common in this repo's own Cargo workspace) showed that intra-workspace dependencies matter for SBOM accuracy.
- **deps.dev enrichment**: the new ecosystems all use PURL types that the existing `deps.dev` enrichment pipeline already supports (per milestone history). No new enrichment-side work is in scope.
- **Test fixture residence**: the new golden-input fixtures live in `mikebom-cli/tests/fixtures/golden_inputs/<ecosystem>/` (in-repo, NOT the external milestone-090 fixtures repo). They are small (< 5 KB each) and tightly coupled to the per-reader implementation.
- **Constitution alignment**: all new annotations (if any) audit per Principle V before introduction. The expected outcome is **zero new `mikebom:*` annotations** — all signal is carried via existing annotations + PURL fields. Lifecycle-scope (`mikebom:lifecycle-scope: "build"`) on Gradle buildscript entries is an existing annotation reused per Principle V.
