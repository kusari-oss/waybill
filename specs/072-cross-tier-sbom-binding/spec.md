# Feature Specification: Cross-tier SBOM binding

**Feature Branch**: `072-cross-tier-sbom-binding`
**Created**: 2026-05-04
**Status**: Draft
**Input**: User description: When a single component (e.g., the binary `foo`) is described by multiple mikebom-emitted SBOMs at different lifecycle tiers — a source-tier SBOM (mikebom scanning the repo), a build-time-trace SBOM, and an image-tier SBOM (mikebom scanning the deployed container image) — there is currently no enforced way for downstream consumers to verify that the *image*'s `foo` is the same `foo` the source/build SBOM describes. PURL identity is necessary but not sufficient: two binaries built from the same source at different times share a PURL but have different content hashes, and a binary in the image may have been pulled in via a path the source SBOM never observed (vendored, base-layer, sidecar). This gap makes cross-tier VEX propagation dangerous: a "not-affected" assertion attached to the source-tier component should only transfer to the image-tier component when there is verifiable evidence the two components are *the same* component — same source, same build inputs, same content. Without that, a real vulnerability in the image (introduced by a different code path) can be masked by a propagated source-tier VEX. The user wants mikebom to (a) emit cross-tier identity bindings strong enough that consumers can verify "image's foo == source's foo", (b) make the binding strength explicit so consumers know whether to trust a propagated VEX, and (c) ideally surface the case "this image-tier component does NOT trace back to a known source-tier SBOM" so consumers see when they're operating on unbound data.

## Clarifications

### Session 2026-05-05

- Q: What goes into the per-component binding hash? → A: Layered hash: composite of VCS commit + lockfile-content-hash + main-module manifest hash, canonically serialized. The hash combines the three sides where all are present (`verified`); lockfile + manifest only (`weak`); fewer than two sides → `unknown`. Survives non-reproducible builds, detects a "different foo" because a different commit / lockfile / manifest produces a different hash. Does NOT require byte-identical reproducible-build output.
- Q: Is per-instance VEX an emission delta in mikebom or a documented consumer-side aggregation convention? → A: Hybrid. Mikebom emits per-instance subjects WITHIN the OpenVEX 0.2.0 schema's existing `Statement.products[]` mechanism — using each component's `bom-ref` (CDX) or `SPDXID` (SPDX) as a supplementary instance identifier alongside the PURL. The OpenVEX schema is NOT extended upstream. Pre-072 OpenVEX consumers continue to receive per-PURL statements as a documented aggregation. Per-instance distinguishability is what makes the worked-example case (US2 AS#4 / SC-003) work end-to-end without requiring an OpenVEX schema fork.
- Q: Where does the VEX-propagation mode live in the CLI? → A: Flag on the existing `mikebom sbom enrich` command — `--vex-propagation-mode {permissive,caveated,strict}` with default `caveated`. The breaking-change opt-out (`--vex-propagation-mode permissive`) is colocated with the behavior it controls. No new subcommand; existing VEX-aware command extends.
- Q: When a source-tier OpenVEX statement has no per-instance identifier (just a PURL) but the target has multiple instances of that PURL, how should propagation behave? → A: Mode-dependent broadcast — `permissive` broadcasts to ALL matching instances unchanged (pre-072 semantic); `caveated` broadcasts to ALL with a `mikebom:vex-binding-status: unverified` caveat on every non-`verified` instance (the C-3 aggregation rule then correctly surfaces unbound instances as `affected`); `strict` propagates ONLY to `verified` instances and refuses the rest with a structured refusal-rationale annotation. Captured in FR-008 and tasks.md T020 explicitly.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — A consumer receives an image SBOM and the source SBOM for the same project; can verify they describe the same binary (Priority: P1) 🎯 MVP

A vulnerability scanner / supply-chain attestation tool / compliance auditor has both the image-tier SBOM (from `mikebom sbom scan --image`) and the source-tier SBOM (from `mikebom sbom scan --path`) for what should be the same project artifact. They want to programmatically assert: the binary `foo` in the image SBOM is *the binary that was built from* the source SBOM's project — not a different `foo` pulled in from somewhere else.

**Why this priority**: This is the foundation for every other cross-tier consumer workflow — VEX propagation, license inheritance, dependency-graph stitching, attestation chains. Without it, every consumer reinvents binding heuristics inconsistently. With it, the binding is canonical and tool-independent.

**Independent Test**: Run mikebom on a project source tree to produce SBOM_A (source-tier). Build the project, package it into an image, and run mikebom on the image to produce SBOM_B (image-tier). Take both SBOMs to a consumer-side verification tool. The tool MUST be able to find a binding statement on the image-tier `foo` component naming the source-tier SBOM (by document hash or document IRI), and MUST be able to verify the binding by computing the source-tier component's content/build-input hash and comparing against the image-tier component's matching hash. Pass = binding verified. Fail = no binding present, or binding present but verification fails.

**Acceptance Scenarios**:

1. **Given** a source-tier SBOM mikebom emitted for project `foo` and an image-tier SBOM mikebom emitted for the image containing `foo`'s output binary, **When** a consumer inspects the image-tier `foo` component, **Then** the component carries a structured `mikebom:source-document-binding` (or equivalent native field) naming the source-tier SBOM document by stable identifier (document IRI / SHA-256) and including a per-component binding hash that ties the binary back to the source-tier project.
2. **Given** the same pair of SBOMs, **When** the consumer fetches the source-tier SBOM and recomputes the binding hash for its `foo` main-module component, **Then** the recomputed hash matches the image-tier binding-hash field exactly.
3. **Given** an image-tier SBOM where binary `foo` was pulled in via a base-layer or sideloaded path mikebom cannot trace to a source SBOM, **When** the consumer inspects `foo`, **Then** the component is explicitly marked as `mikebom:source-document-binding: unknown` (or omits the binding field entirely with an audit-trail annotation), making the unbound state machine-readable.
4. **Given** an image-tier SBOM and a *wrong* source-tier SBOM (e.g., from a different project / different commit), **When** the consumer attempts to verify the binding, **Then** verification fails with a specific reason — the image binding hash does not match what the wrong source SBOM produces.

### User Story 2 — VEX propagation from a source-tier SBOM to an image-tier component is only allowed when the binding is verified (Priority: P1)

The user (or a downstream tool) has propagated a "not-affected, justification: vulnerable_code_not_present" VEX from a source-tier SBOM to the same component appearing in an image-tier SBOM. They want this propagation to be honest: it should fire only if the image-tier component is provably the same component the VEX was originally written against. If the binding is missing, weak, or stale, the propagated VEX MUST carry a "binding-unverified" caveat so the image-tier consumer cannot accidentally treat it as authoritative.

**Why this priority**: This is the user's specific ask. Today VEX propagation is purely PURL-keyed; if the image's `golang.org/x/net@v0.28.0` shows up in source's VEX as not-affected, the not-affected assertion transfers — even if the image's instance came in via a *different* binary that *does* call the vulnerable function. Constraining propagation to the verified-binding case prevents quietly false "not affected" results on image scans.

**Independent Test**: Take a source-tier SBOM with a VEX statement `golang.org/x/net@v0.28.0: not_affected (vulnerable_code_not_present)`. Take an image-tier SBOM that ALSO contains `golang.org/x/net@v0.28.0` but where the image instance was introduced by a different binary mikebom did NOT trace to the source. Run mikebom's VEX-propagation tool. The propagated VEX on the image-tier instance MUST carry a `vex.binding_status: unverified` (or equivalent native field) caveat. The same test with a verified binding (matching content hash + matching VCS commit) must produce the propagated VEX *without* the caveat.

**Acceptance Scenarios**:

1. **Given** a source-tier VEX `not_affected` statement on `pkg:golang/golang.org/x/net@v0.28.0` and an image-tier component `pkg:golang/golang.org/x/net@v0.28.0` whose binding to the source SBOM verifies, **When** the consumer requests "merge VEX from source to image", **Then** the image-tier component receives the `not_affected` statement WITHOUT a binding-unverified caveat, and the propagation step is recorded with provenance pointing back to the source-tier VEX statement and the verified binding.
2. **Given** the same source-tier VEX, but the image-tier component's binding is `unknown` (no source-document binding emitted at all), **When** the consumer requests propagation, **Then** the image-tier component receives the `not_affected` statement with a `binding-unverified` caveat AND a structured rationale stating which binding evidence was missing. Optionally, a strict-mode flag (e.g., `--require-binding`) MUST refuse the propagation outright in this case.
3. **Given** a binding that exists but verification FAILS (image binding-hash does not match source's), **When** the consumer requests propagation, **Then** the propagation MUST be refused (not just caveated) — the binding is actively wrong, and a propagated VEX would be a false safety claim. The tool reports the specific verification mismatch.
4. **Given** a vulnerability that is present in the image via TWO independent component instances (one bound to a source-tier "not-affected" VEX, the other unbound from a base-layer binary), **When** the consumer aggregates the image-tier VEX state for that vulnerability, **Then** the aggregate state is `affected` (because the unbound instance is unverified) — propagating the source-tier VEX onto the bound instance does NOT mask the unbound instance's status. The user's worked example: a Go networking CVE marked not-affected in source because the project doesn't call the vulnerable function, but the image starts a server from non-project code that DOES call it. The aggregate must surface "affected" via the unbound instance even after correctly applying the source-tier VEX to the bound instance.

### User Story 3 — Operator can ask "where did this image-tier component come from?" and get a structured answer (Priority: P2)

A security operator triaging an image-tier component (e.g., "tell me about `pkg:golang/golang.org/x/net@v0.28.0` in this image") wants a machine-readable answer to "which source-tier SBOMs (if any) describe a build that produced this binary?" — and where there are multiple candidate sources (e.g., the same upstream library is bundled by two different first-party binaries), to see all of them.

**Why this priority**: This is the introspection use case behind US1/US2. Without it, operators can't manually audit binding decisions — they have to trust the propagation tool. With it, the binding metadata is a debuggable artifact.

**Independent Test**: An operator has an image-tier SBOM and a directory of candidate source-tier SBOMs. They run a `mikebom sbom trace-binding <component-purl> <image-sbom> <source-sbom-dir>` command. The tool MUST return: (a) a list of source-tier SBOM documents that bind to this component, (b) for each, the binding strength (verified, weak, none), (c) for each, the per-component binding hash + verification status, (d) a one-line summary suitable for inclusion in an audit log.

**Acceptance Scenarios**:

1. **Given** an image-tier SBOM and 3 candidate source-tier SBOMs (one of which actually built this binary), **When** the operator runs the trace-binding lookup, **Then** the matching source SBOM is identified with `binding: verified`; the other 2 are reported with `binding: not-applicable` (their main-module / build outputs don't include this binary).
2. **Given** an image-tier component with NO source-tier match in the candidate set, **When** the lookup runs, **Then** the result is `binding: unknown — no candidate source SBOM contains this component as a build output`. This is a *useful* result, not an error — it tells the operator the image contains code mikebom can't trace.
3. **Given** a component appearing in the image via two paths (the project's first-party binary AND a base-layer system binary), **When** the lookup runs, **Then** BOTH instances are returned with their respective binding states; the operator can see "instance 1: bound to source SBOM X, instance 2: unbound, base-layer source `dpkg/golang-libs`."

### User Story 4 — Future scans automatically propagate binding metadata to downstream tiers (Priority: P3)

When mikebom emits a build-time-trace SBOM after observing a build that consumed a previously-emitted source-tier SBOM, mikebom can include the source-tier SBOM's identifier + the project's main-module binding hash directly into the build-tier SBOM's metadata. Same for image-tier scans that discover binaries built by an observed build.

**Why this priority**: This is the "first-party emit" path. Today consumers have to do the binding work themselves; with this, mikebom does it at emission time and consumers just verify. Lower priority because manual / consumer-side binding (US1) covers the immediate need; first-party emit is a follow-on optimization.

**Independent Test**: Run a full pipeline: source scan → build trace → image scan, where each step has access to the prior tier's SBOM. Inspect the image-tier SBOM. Each first-party component should already carry a `mikebom:source-document-binding` field naming the source-tier SBOM and the build-tier SBOM, with the binding hash pre-computed by mikebom rather than supplied by the consumer.

**Acceptance Scenarios**:

1. **Given** a source-tier SBOM available at a known path / IRI, **When** mikebom is invoked with `mikebom sbom scan --image <ref> --bind-to-source <path-or-iri>`, **Then** every image-tier component whose binding hash matches a source-tier component automatically carries the `mikebom:source-document-binding` metadata in the output.
2. **Given** the same invocation but where some image-tier components do NOT match any source-tier component, **Then** those components carry `mikebom:source-document-binding: unknown` (explicit) and a transparency annotation indicating the consumer-binding lookup was attempted but no match was found.

### Edge Cases

- **Binary identical bytes from two different source codebases.** Possible when reproducible-build pipelines produce byte-identical binaries from two repos with shared subset content (rare). The binding hash matching reports `binding: verified` against either source SBOM — the consumer must understand that "verified" means content-equivalent, not provenance-unique. Documented: the mikebom binding statement asserts content equivalence, NOT exclusive provenance. For cryptographic provenance, layer in-toto attestations on top.

- **Source SBOM is updated after image is built.** Common in CI: the source SBOM is regenerated whenever the source tree changes, but the image SBOM is pinned to a specific build moment. The binding MUST tolerate "source SBOM at commit X, image at commit X" — the linkage is by commit/content, not by document timestamp. If the source SBOM is regenerated for the same commit, its document hash changes but the per-component binding hashes (which are commit + module-set derived) stay stable; binding still verifies.

- **An image binary is statically linked from many source projects (e.g., a Go binary embedding multiple Go modules from different repos).** Mikebom emits one main-module per source-tree project. The image binary's `mikebom:source-document-binding` SHOULD be a *list* of bindings — one per source SBOM that contributed code — rather than a single scalar. Each entry has its own verification state.

- **Source SBOM exists but mikebom has no observability into the build step.** The source SBOM describes what's on disk; the build is opaque (no eBPF trace was captured). The binding from image → source is then "necessary but not sufficient" — content match is plausible but not proven. Catalogue this as `binding: weak (source-content-match, build-not-traced)`. Distinct from `verified` (which requires traced build).

- **Per-instance VEX rather than per-coord.** When the same coord (e.g., `pkg:golang/golang.org/x/net@v0.28.0`) appears in the image via two independent paths (a verified-bound first-party binary and an unbound base-layer binary), VEX state MUST be per-instance — one instance can be `not_affected` (with verified binding) while the other remains `affected` (unbound). Aggregating to a single per-coord VEX would mask the unbound vulnerability and is exactly the user-described failure mode.

- **The binding hash computation must be deterministic across mikebom versions.** A binding written by mikebom alpha.14 must verify against a binding written by alpha.15 if the underlying inputs (source content, module list, VCS commit) are identical. The hash algorithm + input canonicalization is part of the spec contract; changing it requires a versioned binding scheme (V1 → V2) with parallel emission for a deprecation window.

- **Consumer with a partial source SBOM (e.g., excluded scopes).** If the source SBOM was generated with `--exclude-scope dev,build,test`, its main-module binding hash is computed over the surviving component set. The image-tier binding (which sees runtime deployment) might match the exclude-scope source SBOM but NOT a full-include source SBOM. Document: the binding is meaningful within a `--exclude-scope` configuration; mismatched configs produce a verifiable failure (and the failure rationale should name the scope-config delta).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Mikebom MUST emit a structured cross-tier binding annotation on every component it produces in a non-source tier (`build`, `deployed`) that ties the component back to the source-tier SBOM (or build-tier SBOM) that describes its origin. The binding MUST include: (a) source SBOM document identifier (SHA-256 or stable IRI), (b) per-component binding hash that ties this specific component to the source-tier project's main-module hash, (c) binding-strength label (`verified`, `weak`, `unknown`).
- **FR-002**: The per-component binding hash MUST be a **layered composite** of three input sides, canonically serialized and hashed via SHA-256: (1) **VCS commit identity** — the project's source-control commit identifier (Go BuildInfo `vcs.revision`, git HEAD SHA via `git rev-parse`, or equivalent for the source ecosystem); (2) **lockfile content hash** — SHA-256 of the project's lockfile bytes (`Cargo.lock`, `package-lock.json`, `Gemfile.lock`, `go.sum`, `poetry.lock`, etc.) selected per ecosystem; (3) **main-module manifest hash** — SHA-256 of the project's top-level manifest (`Cargo.toml`, `package.json`, `pom.xml`, `*.gemspec`, `pyproject.toml`, `go.mod`) after canonical normalization (whitespace + key-order). The three sides are concatenated as a JSON object `{"vcs":"<commit-or-null>","lockfile":"<sha256-or-null>","manifest":"<sha256-or-null>","algo":"v1"}` (sorted keys, no whitespace) and SHA-256'd. The binding hash MUST be deterministic and reproducible — given the same source-tier inputs, two mikebom invocations MUST produce the same binding hash; determinism MUST hold across mikebom alpha versions for the same `algo` version.
- **FR-003**: When mikebom cannot determine the source-tier SBOM for an image-tier component (e.g., base-layer system package, sideloaded binary), it MUST emit an explicit `binding: unknown` marker on the component rather than silently omitting the binding. The marker MUST include a rationale (e.g., `reason: base-layer-system-package`).
- **FR-004**: The binding metadata MUST be standards-native where each format provides one (CycloneDX `externalReferences[type: bom]` for cross-document refs, SPDX `externalDocumentRefs` + relationship type `BUILT_FROM` / `GENERATED_FROM`, SPDX 3 `Relationship.relationshipType: built_from`). Where standards-native fields cannot express the per-component binding hash or strength label, mikebom MUST emit a `mikebom:source-document-binding` annotation per Constitution Principle V — but the standards-native cross-document reference MUST be present alongside.
- **FR-005**: Mikebom MUST provide a consumer-side verification command (e.g., `mikebom sbom verify-binding <image-sbom> <source-sbom>`) that recomputes the per-component binding hash from the source-tier SBOM and compares against the image-tier binding-hash assertion. Output MUST be machine-parseable (JSON) and report per-component pass/fail with verification rationale.
- **FR-006**: Mikebom MUST provide an introspection command (`mikebom sbom trace-binding <component-purl> <image-sbom> <source-sbom-or-dir>`) that, given an image-tier component, returns the candidate source-tier SBOMs and their binding state. Useful for operator triage.
- **FR-007**: Mikebom's existing VEX propagation logic in the `mikebom sbom enrich` command MUST be extended to require — or visibly caveat — propagation across tier boundaries based on binding state. Propagation is controlled by a new `--vex-propagation-mode {permissive,caveated,strict}` flag on `mikebom sbom enrich`: (a) `permissive` (pre-072 behavior — propagate by PURL identity, no binding check), (b) `caveated` (propagate but tag with `binding-unverified` when verification fails or is absent), (c) `strict` (refuse propagation when binding is not `verified`). The flag's default value flips to `caveated` in this milestone — a documented breaking change. Pre-072 callers needing exact prior behavior pass `--vex-propagation-mode permissive`. The flag lives on the existing VEX-aware command rather than a new subcommand.
- **FR-008**: When the same PURL appears in an image-tier SBOM via two independent component instances (different `bom-ref` values in CDX, different `SPDXID` values in SPDX), each instance MUST be addressable by its own VEX statement. Mikebom's emitted OpenVEX 0.2.0 sidecar MUST identify per-instance subjects using the existing `Statement.products[]` array — each affected product entry MUST carry both the PURL and a supplementary instance identifier (`bom-ref` for CDX-paired SBOMs, `SPDXID` for SPDX-paired SBOMs) in the product's identifier set. Pre-072 consumers that ignore the supplementary identifier receive a per-PURL aggregation as before; post-072 consumers that recognize it can apply VEX statements at instance granularity. The OpenVEX schema is NOT extended upstream — instance identifiers travel as additional values in the existing identifier-set field. The aggregation rule for the per-PURL fallback (`affected ⊕ unbound-and-not-explicitly-vexed = affected`) MUST be documented in `docs/reference/cross-tier-binding.md`. **Propagation corner-case (per 2026-05-05 clarification F2):** when a source-tier OpenVEX statement carries no per-instance identifier (just a PURL — typical of pre-072 OpenVEX input) but the target SBOM has multiple instances of that PURL, propagation behavior is mode-dependent: (a) `permissive` → broadcast the statement to ALL matching instances (pre-072 semantic preserved); (b) `caveated` → broadcast to ALL matching instances WITH `mikebom:vex-binding-status: unverified` on every instance whose binding strength is not `verified` (the C-3 aggregation rule then surfaces unbound instances as `affected` regardless); (c) `strict` → propagate ONLY to instances with `verified` binding strength; refuse the rest with a structured refusal-rationale annotation per FR-007.
- **FR-009**: When propagated VEX statements carry the `binding-unverified` caveat, the caveat MUST be machine-readable in the output (e.g., a `vex.binding_status` field or an OpenVEX `justification` extension) AND human-readable in the propagation rationale (so operators reading the SBOM can see why a VEX is not authoritative).
- **FR-010**: The binding annotation contract (algorithm version, input set definition, hash format) MUST be documented in `docs/reference/cross-tier-binding.md` such that an external implementer can write a verifier compatible with mikebom's emission. Same standard as the milestone-071 conformance-harness-author guide.
- **FR-011**: When mikebom is invoked with `--bind-to-source <path-or-iri>` (or equivalent), it MUST attempt to resolve the source-tier SBOM at the named location and pre-compute the binding metadata into the emitted output. If the source-tier SBOM cannot be loaded or doesn't contain the expected components, the failure MUST be reported (per Constitution Principle X transparency); the scan MUST NOT silently emit components without binding when binding was requested.
- **FR-012**: The binding-strength label is determined by which of FR-002's three input sides are present and match the source-tier SBOM's recomputation: (a) `verified` — all three sides (VCS commit + lockfile-content-hash + manifest-hash) are present in BOTH the source-tier and image-tier evidence AND match; (b) `weak` — exactly two of the three sides are present and match (most commonly: lockfile + manifest match but VCS commit was not captured by the source-tier scan, e.g., scan ran outside a git checkout); (c) `unknown` — fewer than two sides are present, or any present side fails to match the source-tier recomputation. Components where the image-tier evidence carries an additional content-hash match against the source-tier expected-output hash (e.g., reproducible-build pipelines that publish a deterministic binary SHA-256 in the source SBOM) MAY be additionally labeled `verified+content-hash` as a strict superset of `verified`, but this stricter label is OPTIONAL and consumers MUST treat it as equivalent to `verified` for default policy gating.

### Key Entities

- **Source-tier SBOM**: An SBOM mikebom emitted via `mikebom sbom scan --path <dir>` describing a project's source tree state. Carries `mikebom:sbom-tier: source` and a project main-module component (per milestones 053–070).
- **Build-tier SBOM**: An SBOM mikebom emitted via `mikebom trace` describing build-time observed state. Carries `mikebom:sbom-tier: build`. Out of scope for the MVP if eBPF traces aren't broadly available; the binding contract supports it but US1's MVP is source ↔ image.
- **Image-tier SBOM**: An SBOM mikebom emitted via `mikebom sbom scan --image <ref>` describing an image's deployed state. Carries `mikebom:sbom-tier: deployed`.
- **Cross-tier binding**: A structured assertion that an image-tier (or build-tier) component originates from a specific source-tier (or earlier-tier) SBOM. Has three components: source SBOM identifier, per-component binding hash, binding-strength label.
- **Binding hash**: A deterministic hash over the source-tier inputs that produced the component. Algorithm version is part of the contract.
- **Binding strength**: One of `verified` (cryptographic content match), `weak` (PURL + VCS-commit match without content hash), `unknown` (no source-tier evidence).
- **VEX propagation mode**: One of `permissive` (default for back-compat), `caveated` (propagate with binding-unverified tag when needed), `strict` (refuse unverified propagations).
- **Per-instance VEX state**: Each component instance (each bom-ref / SPDXID) carries its own VEX assertion, distinct from any other instance of the same PURL.
- **Aggregate VEX state**: A tool-computable rollup across instances of the same PURL, governed by the FR-008 aggregation rule.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A consumer with an image-tier SBOM and a source-tier SBOM for the same project can verify the cross-tier binding for every first-party component (≥95% of the image-tier components mikebom claims trace to source) without needing to re-run any mikebom scan; a single command returns pass/fail per component.
- **SC-002**: When mikebom propagates a VEX statement from source-tier to image-tier in `caveated` mode, 100% of propagated statements that lack a verified binding carry the `binding-unverified` caveat; 0% of verified-binding propagations carry the caveat unnecessarily.
- **SC-003**: In the worked-example scenario (source-tier `not_affected` VEX on `pkg:golang/golang.org/x/net@v0.28.0`, image with same coord present via two paths — one bound to source, one unbound from base-layer), the aggregate image-tier vulnerability state for the CVE in question reports `affected` (because the unbound instance defaults to "could be affected") even after correctly applying the source-tier VEX to the bound instance. The user can read the SBOM and clearly distinguish "instance A: not-affected (source-VEX)" from "instance B: status unknown (no binding)".
- **SC-004**: An external auditor can write a verifier from `docs/reference/cross-tier-binding.md` alone (no mikebom source code required) and successfully validate ≥95% of bindings on a published reference fixture set.
- **SC-005**: When mikebom is invoked with `--bind-to-source <path>` and the source SBOM exists, 100% of image-tier components for which mikebom has the necessary source-tier evidence carry a binding annotation with non-`unknown` strength label.
- **SC-006**: When mikebom cannot trace an image-tier component to any source-tier SBOM, the resulting `binding: unknown` marker carries a structured `reason` that names *why* the trace failed (≥95% of unknown markers have a non-empty reason).
- **SC-007**: The binding hash algorithm is deterministic across mikebom alpha versions: a binding written by alpha.14 verifies under alpha.15 when source inputs are byte-identical.
- **SC-008**: Existing mikebom workflows that don't use cross-tier binding (the alpha.14 baseline) continue to work without modification — the binding emission is additive, the new `--bind-to-source` flag is opt-in for `--image` scans, and the default VEX-propagation mode change to `caveated` is documented as a breaking change in the milestone-072 release notes with a `--vex-propagation-mode permissive` opt-out for callers needing pre-072 behavior.

## Assumptions

- mikebom's existing per-component evidence model (PURL, hashes, VCS metadata, deep-hash per-file SHA-256s from milestone 038) provides enough raw signal to compute a meaningful binding hash for the common cases. Where it doesn't (e.g., a source SBOM with no main-module — pre-053 Go scans before #104 closure), binding falls back to `weak` or `unknown` rather than failing.
- The standards-native cross-document reference fields (CDX `externalReferences[type: bom]`, SPDX `externalDocumentRefs`) are sufficient carriers for the source SBOM identifier. Per-component binding hash + strength label are mikebom-specific signal carried via `mikebom:source-document-binding` annotation per Constitution Principle V (no native standards equivalent at the per-component level today).
- OpenVEX 0.2.0's existing structure can carry the binding-unverified caveat via the `justification` or status-extension fields. If a richer per-statement extension is needed, the work would be split into a separate spec for OpenVEX-side schema work — out of scope for this milestone.
- Per-instance VEX state means each `bom-ref` / `SPDXID` carries its own VEX. Existing mikebom emission already produces unique bom-refs per instance (cross-format-parity-checked by milestone 071); per the 2026-05-05 clarification this milestone DOES extend mikebom's OpenVEX 0.2.0 sidecar emission to put those instance identifiers into the `Statement.products[]` array alongside the PURL. The OpenVEX schema is NOT extended upstream — instance identifiers travel as additional values in the existing identifier-set field. Pre-072 OpenVEX consumers ignore the supplementary identifier and see per-PURL aggregation; post-072 consumers can apply per-instance VEX.
- The VEX-propagation default change (`permissive` → `caveated`) IS a breaking change for users who depend on the current quiet propagation. The milestone release notes call this out explicitly; `--vex-propagation-mode permissive` provides the opt-out.
- Build-tier SBOM bindings (US4 case) are valuable but follow the same contract as source-tier bindings — implementing source ↔ image binding (US1) in the MVP automatically extends to build ↔ image with no additional spec work.
- Reproducible-build infrastructure is *not* required to use mikebom binding — but where it exists, mikebom's `verified` label is meaningful (the source SBOM's expected build-output hash matches the image's actual content). Where reproducibility isn't available, `weak` is the best label achievable for non-traced builds.
- Consumers have access to the source-tier SBOM document at the time they verify the binding. mikebom may include a SHA-256 hash of the source SBOM document so consumers can confirm they're reading the right one; mikebom does NOT host the source SBOMs (no registry / no fetch).

## Out of Scope

- **Mikebom does not introspect whether a vulnerable function is actually called in the image binary.** That's static-analysis territory (callgraph reachability), not observation. mikebom's role is binding evidence; whether a propagated VEX is *semantically* correct in a consumer's risk model is the consumer's call.
- **A new VEX format or OpenVEX schema extension.** mikebom's binding caveat lives in the existing OpenVEX justification field or a sidecar mikebom-specific annotation. No upstream OpenVEX work in this milestone.
- **Networked / hosted SBOM registry for binding lookup.** mikebom emits the source SBOM identifier; consumers (or external CI infrastructure) handle the actual fetch/cache.
- **Cross-vendor binding** (e.g., binding a vendor-supplied SBOM to a downstream image scan). Same contract should apply, but the MVP focuses on first-party (mikebom-source ↔ mikebom-image) bindings to keep the test surface bounded. Vendor-side bindings are a follow-on.
- **Binding-strength upgrade from external evidence.** Consumers may have additional provenance (in-toto attestations, signed build records) that strengthens a binding from `weak` to `verified`. mikebom does not consume those; it emits its own evidence and lets consumers compose.
- **Format-version migrations.** Targets remain CDX 1.6 / SPDX 2.3 / SPDX 3.0.1 — same as alpha.14.
- **Backfilling binding metadata onto pre-072 mikebom-emitted SBOMs.** A separate `mikebom sbom enrich-binding` tool is conceivable but out of scope. New scans on alpha.15+ get the binding emission natively.
