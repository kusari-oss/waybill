## T005 — name → PURL resolution multi-version stdlib disambiguation

Verified at `mikebom-cli/src/scan_fs/mod.rs:498-548` + resolution at
line 756-772. `name_to_purl: HashMap<(String, String), String>` keyed
by (ecosystem, name). For multi-Go-version scans (multi-binary image
where 2 Go binaries use different Go stdlib versions), both stdlib
entries have name="stdlib" + ecosystem="golang" → second insert
overwrites the first. Only ONE stdlib PURL is resolvable via name-
lookup.

Decision: accept degraded behavior for the multi-Go-version edge case
(all Go mainmods link to whichever stdlib entry name_to_purl happens
to hold at emission time). Single-version scans (pico's case + the
common case) work correctly. Multi-version disambiguation is out of
scope for m194; if it surfaces as a real customer issue, follow-up
fix would extend `name_to_purl` value type from `String` → `Vec<String>`
or use `(ecosystem, name, version)` as the composite key.

FR-003 acceptance test in the spec explicitly permits "if only one
stdlib is emitted, all Go mainmods link to it" — my implementation
matches this.
