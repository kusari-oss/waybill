# Feature Specification: Kotlin + Swift Ecosystem Readers

**Feature Branch**: `122-kotlin-swift-readers`
**Created**: 2026-06-14
**Status**: Draft
**Input**: User description: "I would like to explore essentially following similar patterns as we already have for Kotlin and Swift" — clarified during /speckit-specify to mean: add BOTH new readers (Kotlin-DSL Gradle + Swift Package Manager) parallel to the existing per-ecosystem readers (cargo, npm, pip, gem, maven, gradle, nuget, …).

## Context

mikebom's filesystem package-detection layer (`scan_fs/package_db/`) today covers ten mainstream ecosystems — apk, cargo, deb, gem, golang, maven, npm, pip, nuget, rpm, yocto — plus the milestone-106 Gradle / uv / bun / nuget expansion. Two meaningful gaps remain for mobile-and-Apple developer cohorts:

| Issue | Ecosystem | Today's behavior | Coverage gap |
|---|---|---|---|
| (new) | **Kotlin / Gradle Kotlin DSL** | The milestone-106 Gradle reader parses `gradle.lockfile` (text format) only. Kotlin-DSL build sources (`build.gradle.kts`, `settings.gradle.kts`, Kotlin Multiplatform layout) and version-catalog files (`libs.versions.toml`) are NOT mined for dependency declarations. | Projects using Kotlin-DSL without `gradle.lockfile` enabled (the default Android Studio + IntelliJ project shape) produce empty Java SBOMs from source-tree scans, even when their `build.gradle.kts` declares dozens of dependencies. |
| (new) | **Swift Package Manager (SPM)** | Zero components emitted from any Swift / Apple developer project. No `Package.swift` parser; no `Package.resolved` parser. Mach-O binary analysis exists at the binary layer but does not surface SPM dependencies. | Every Swift / iOS / macOS project scanned at the source-tree level produces an empty SBOM. |

This milestone closes both gaps following the established ecosystem-reader pattern: new `scan_fs/package_db/<ecosystem>/` modules slot into the existing `read_all` dispatcher; emit PURLs in already-wired ecosystems where possible (`pkg:maven/...` for Kotlin-via-Gradle; `pkg:swift/...` per the [purl-spec swift type](https://github.com/package-url/purl-spec/blob/main/PURL-TYPES.rst#swift)); reuse the existing `PackageDbEntry` shape; integrate via existing deduplication + annotation pipelines.

The two ecosystems ship under one coordinated milestone because:

- They fit the **same shape of work** — declarative file parsing → existing `PackageDbEntry` → existing dispatcher.
- They serve overlapping operator segments — mobile developers, Apple-stack developers, KMP (Kotlin Multiplatform) teams that ship both Android + iOS from one repo.
- The operator-facing docs (`docs/ecosystems.md`, the coverage matrix) get one coherent update.
- Each user story still ships **independently** so the milestone has natural cut-points.

## Clarifications

### Session 2026-06-15

- Q: For a `Package.resolved` entry pinned by commit only (no `state.version`), what goes in the `@<version>` segment of the emitted `pkg:swift/...` PURL? → A: Full 40-char revision SHA as version (matches Go reader's `pkg:golang/...@<full-sha>` convention; preserves uniqueness without re-encoding work for consumers).
- Q: When a Kotlin Multiplatform dep is declared in multiple source-sets, how does mikebom emit the `mikebom:kmp-source-set` annotation? → A: ONE component per dep PURL; `mikebom:kmp-source-set` value is a JSON-encoded ARRAY of every source-set name that declared it (mirrors the milestone-116 C64 `mikebom:produces-binaries` + milestone-119 C67 `mikebom:assertion-conflict` JSON-array storage convention; preserves canonical PURL uniqueness).
- Q: Does v0.1 parse `Package.swift` for dependency declarations (local-path deps, workspace members, target declarations)? → A: No — v0.1 reads `Package.resolved` only. `Package.swift` presence is detected (signals "this is a SwiftPM project root") but never parsed for content. Local-path / workspace-member emission, target declarations, and conditional-platform deps are all deferred to a future phase. Resolves contradiction between FR-002 and the original US1 AS5.
- Q: What PURL shape does the synthetic Kotlin/Gradle workspace-root component carry? → A: `pkg:generic/<rootProject.name>@0.0.0` (matches the milestone-106 uv / bun workspace-root convention used uniformly across every existing multi-module reader; `pkg:generic/` is the project's standing container for "identity exists but no published ecosystem maps to it").
- Q: How are `build.gradle.kts`-only-discovered components (no `gradle.lockfile`, no installed JARs) classified for tier + visibility? → A: Design-tier (`mikebom:sbom-tier = "design"`), gated by the existing `--include-declared-deps` flag. Matches milestone-106's Maven `pom.xml`-only and pip `requirements.txt` precedent: `--path` scans get them auto-on (the "what would be pulled in on build" view); `--image` scans get them only when explicitly opted in. Preserves artifact-tier faithfulness on image scans while keeping the source-tree-scan promise (SC-004).

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Swift / Apple developer project with Package.resolved (Priority: P1)

A Swift developer has an iOS app, a macOS CLI, or a server-side Swift project. The project root has a `Package.swift` (the SwiftPM manifest) and a `Package.resolved` (the SwiftPM lockfile — JSON, deterministic, version-pinned). The developer runs `mikebom sbom scan --path .` against the source tree and expects every dependency from `Package.resolved` to appear as an SBOM component with the resolved version + commit-pinned source location.

**Why this priority**: Swift is the entire Apple-platform-developer cohort plus a growing server-side Swift community (Vapor, Hummingbird, AWS Lambda runtime, swift-nio-based infrastructure). mikebom today produces empty SBOMs for every one of these projects. `Package.resolved` is the smallest possible unlock — a single deterministic JSON file at the repo root with the same shape across all SwiftPM versions since Swift 5.0. Implementation parallels the existing lockfile-driven readers (cargo, npm, pip, gem); no new resolver, no enrichment changes.

**Independent Test**: Add a fixture `mikebom-cli/tests/fixtures/golden_inputs/swift_package_resolved/` containing a `Package.swift` declaring 2–4 deps (e.g., `swift-argument-parser`, `swift-log`, `Alamofire`) plus its corresponding `Package.resolved` (the canonical version-2 JSON shape). Scan the fixture; assert each dep emerges as a `pkg:swift/<host>/<namespace>/<name>@<version>` PURL with `mikebom:source-files: ".../Package.resolved"` and commit-pinned `mikebom:source-revision` annotation surfacing the lockfile's `pins[].state.revision` hash.

**Acceptance Scenarios**:

1. **Given** a project with `Package.resolved` declaring `swift-argument-parser` at version `1.3.0` pinned to commit `abc123...`, **When** scanned, **Then** an SBOM component `pkg:swift/github.com/apple/swift-argument-parser@1.3.0` emerges with `mikebom:source-files: ".../Package.resolved"`.
2. **Given** a `Package.resolved` entry whose `location` URL is `https://github.com/Alamofire/Alamofire.git` (with the `.git` suffix the SwiftPM lockfile encodes), **When** scanned, **Then** mikebom strips the `.git` suffix when projecting onto the PURL `namespace`/`name` segments so the resulting PURL matches the canonical `pkg:swift/github.com/Alamofire/Alamofire@<ver>` shape recognized by the purl-spec examples.
3. **Given** a `Package.resolved` entry pinned by commit-only (`state.version` absent, `state.revision` present — the SwiftPM "branch-tracking" mode), **When** scanned, **Then** mikebom emits the component with the FULL 40-char revision SHA as the PURL version segment (`pkg:swift/<host>/<ns>/<name>@<full-sha>`) and `mikebom:source-type: "git"` annotation. The same SHA also appears verbatim on the `mikebom:source-revision: "<sha>"` annotation for grep-friendly cross-reference.
4. **Given** a `Package.swift` manifest without a `Package.resolved` (a fresh checkout pre-resolve), **When** scanned, **Then** mikebom emits an explicit `tracing::warn!` naming the gap ("Package.swift found without Package.resolved; run `swift package resolve` to lock dependencies") and emits ZERO components from this file — the manifest-only path is design-tier and out of scope for v0.1 (matches the cargo no-lockfile-no-deps pattern).

### User Story 2 — Android / Kotlin project with build.gradle.kts (Priority: P1)

A developer has an Android app or a Kotlin Multiplatform (KMP) library. The project root has `build.gradle.kts` (Kotlin-DSL build script — the default for new Android Studio + IntelliJ projects since 2023) instead of the legacy Groovy `build.gradle`. The build also commonly carries a `gradle/libs.versions.toml` version catalog (the Gradle 7+ convention for centralized version management). There is NO `gradle.lockfile` because dependency locking is not enabled (the project-template default). The developer runs `mikebom sbom scan --path .` and expects the declared dependencies to appear as `pkg:maven/...` SBOM components.

**Why this priority**: Every new Android project created since 2023 uses Kotlin-DSL by default. The same is true for IntelliJ Kotlin/Java projects. mikebom's existing Gradle reader (milestone 106) handles `gradle.lockfile` only — which most projects don't generate. The result: most Android / Kotlin projects produce empty SBOMs from source-tree scans today. P1 because the affected cohort is enormous (every Kotlin / Android developer who hasn't manually enabled dependency locking).

**Independent Test**: Add a fixture `mikebom-cli/tests/fixtures/golden_inputs/kotlin_dsl_gradle/` with a `build.gradle.kts` declaring `dependencies { implementation("com.squareup.okhttp3:okhttp:4.12.0"); api("org.jetbrains.kotlin:kotlin-stdlib:1.9.20") }` plus a `gradle/libs.versions.toml` declaring versioned library references that the build script consumes via `libs.okhttp` syntax. Scan; assert both declared deps emerge as `pkg:maven/<group>/<name>@<version>` components.

**Acceptance Scenarios**:

1. **Given** a project with `build.gradle.kts` declaring `implementation("com.squareup.okhttp3:okhttp:4.12.0")`, **When** scanned, **Then** an SBOM component `pkg:maven/com.squareup.okhttp3/okhttp@4.12.0` emerges with `mikebom:source-files: ".../build.gradle.kts"`.
2. **Given** a `gradle/libs.versions.toml` declaring `[versions] okhttp = "4.12.0"` + `[libraries] okhttp = { module = "com.squareup.okhttp3:okhttp", version.ref = "okhttp" }` AND a `build.gradle.kts` referencing `libs.okhttp`, **When** scanned, **Then** mikebom resolves the version catalog reference and emits `pkg:maven/com.squareup.okhttp3/okhttp@4.12.0` (same PURL as if the dep were declared inline).
3. **Given** a `build.gradle.kts` with multiple dep configurations (`implementation(...)`, `api(...)`, `testImplementation(...)`, `kapt(...)`), **When** scanned, **Then** mikebom emits each dep with the appropriate `mikebom:lifecycle-scope` annotation: `implementation` / `api` → runtime (no annotation); `testImplementation` → `test`; `androidTestImplementation` → `test`; `debugImplementation` → `development`; `kapt` / `annotationProcessor` → `build`. Maps consistently with the milestone-106 `buildscript-gradle.lockfile` lifecycle-scope mapping.
4. **Given** a Kotlin Multiplatform layout (`build.gradle.kts` with `kotlin { jvm(); iosX64(); android() }` blocks declaring dependencies in `commonMain`, `jvmMain`, `iosX64Main` source-sets), **When** scanned, **Then** mikebom emits each declared dependency EXACTLY ONCE per canonical PURL, with `mikebom:kmp-source-set` set to a JSON-encoded ARRAY of every source-set name that declared it (e.g., `"[\"commonMain\",\"jvmMain\"]"` for a dep declared in both). The component-once-per-PURL invariant preserves dedup; consumers parse the JSON array to filter to a specific target. This mirrors the C64 `mikebom:produces-binaries` + C67 `mikebom:assertion-conflict` JSON-array storage convention.
5. **Given** a `settings.gradle.kts` declaring multi-module projects (`include(":app", ":lib")`), **When** scanned, **Then** each module's `build.gradle.kts` is parsed and emitted as a workspace member with `mikebom:component-role: "main-module"`, consistent with the milestone-106 workspace pattern. A synthetic workspace-root component derived from the root project's `rootProject.name` (falling back to the workspace directory name when `rootProject.name` is absent, per FR-007 + clarification Q4) carries `mikebom:component-role: "workspace-root"`.
6. **Given** a `build.gradle.kts` that uses dynamic version resolution (`implementation("io.example:lib:+")` for "latest", `implementation("io.example:lib:[1.0, 2.0)")` for a range), **When** scanned, **Then** mikebom emits the dep with `version` preserved verbatim (`"+"`, `"[1.0, 2.0)"`) and `mikebom:source-type: "design"` to mark it as design-tier (unlocked) — same pattern as `requirements.txt` open ranges.

### User Story 3 — Kotlin Multiplatform repo shipping iOS + Android from one source tree (Priority: P2)

A team ships an iOS app + an Android app + a shared KMP library from one monorepo. The repo contains:

- One root `build.gradle.kts` + `settings.gradle.kts` (Gradle workspace orchestration)
- `androidApp/` — an Android module with `build.gradle.kts` declaring Android-side deps (Compose, Hilt, Retrofit)
- `iosApp/` — an iOS app shell with `Package.swift` declaring iOS-side deps (Alamofire, swift-log)
- `shared/` — a KMP library with `build.gradle.kts` declaring `commonMain`, `iosMain`, `androidMain` dependencies + a published Swift Package wrapper

The developer runs `mikebom sbom scan --path .` against the repo root and expects **one** SBOM that surfaces both ecosystems: `pkg:maven/...` components for the Android side AND `pkg:swift/...` components for the iOS side, with dependency edges preserving which side declared what.

**Why this priority**: KMP is the meaningful use case where having BOTH new readers in one milestone (vs two separate ones) creates compounding value. The single-source-tree polyglot scan is what mikebom does well today for Cargo / Go / Python; extending the model to Kotlin + Swift makes mikebom's "scan a polyglot monorepo with no per-ecosystem config" promise real for the mobile cohort.

**Independent Test**: Vendor a minimal KMP fixture under `mikebom-cli/tests/fixtures/golden_inputs/kmp_polyglot/` with three modules (`androidApp/`, `iosApp/`, `shared/`). Scan the root; assert the resulting SBOM contains the Android-side `pkg:maven/...` set AND the iOS-side `pkg:swift/...` set with no spurious cross-ecosystem dedup collapses (a Maven coord and a Swift coord with the same name are distinct PURLs and emit as distinct components).

**Acceptance Scenarios**:

1. **Given** the KMP monorepo fixture, **When** scanned, **Then** the emitted SBOM contains both `pkg:maven/...` and `pkg:swift/...` PURLs simultaneously; component count for each side matches the per-module ground truth.
2. **Given** the KMP shared module's `commonMain` declares `kotlinx.serialization`, **When** scanned, **Then** the dep appears exactly once with `mikebom:kmp-source-set` value `"[\"commonMain\"]"` (JSON-encoded array) AND a workspace-root → shared → kotlinx.serialization dependency edge.
3. **Given** the iOS module's `Package.swift` declares `Alamofire 5.9.0`, **When** scanned, **Then** the dep appears with PURL `pkg:swift/github.com/Alamofire/Alamofire@5.9.0` AND a workspace-root → iosApp → Alamofire dependency edge.
4. **Given** both modules independently declare `swift-log` (Android side via a Maven-published KMP binding; iOS side via SPM), **When** scanned, **Then** mikebom emits TWO distinct components — `pkg:maven/.../kotlin-logging-jvm@...` and `pkg:swift/github.com/apple/swift-log@...` — because PURLs in different ecosystems are distinct entities. NO spurious dedup collapse on name similarity.

### Edge Cases

- **`Package.swift` references a private Git host via SSH** (`git@gitlab.acme.com:internal/lib.git`): mikebom MUST emit the PURL using the SSH host segment (`pkg:swift/gitlab.acme.com/internal/lib@<ver>`), strip the `.git` suffix, and NOT make a network request to confirm the URL is reachable. Operators on air-gapped CI should see the component appear with the air-gapped host in the namespace.
- **`Package.resolved` schema version drift**: SwiftPM has shipped `version: 1`, `version: 2`, and `version: 3` of the lockfile JSON shape across Swift 5.0–6.0. mikebom MUST parse all three by reading the `version` integer at the top of the file and dispatching to the matching schema; unknown versions warn-and-continue.
- **`build.gradle.kts` uses a `buildSrc/` convention plugin or Kotlin `apply(from = ...)` to factor out dep declarations**: mikebom MUST follow at most one level of `apply(from = "common.gradle.kts")` indirection. Recursive convention-plugin chains (rare) emit a `tracing::warn!` and stop walking; the operator can manually inline-declare the common deps. The mikebom-101 walker patterns already enforce a safety bound here.
- **`libs.versions.toml` references a version that's NOT in the catalog** (typo or stale catalog): mikebom MUST emit a `tracing::warn!` naming the missing key and the referencing file, then drop the component (vs emitting with a placeholder version) so consumers downstream don't see fake versioned components.
- **Mixed Groovy + Kotlin DSL in one project** (legacy `build.gradle` at root, modern `build.gradle.kts` in modules): mikebom MUST parse both, dedup by canonical PURL, and prefer the locked-via-gradle.lockfile version when both are present (existing milestone-106 dispatcher behavior — extended to recognize `build.gradle.kts` as a discovery source).
- **Swift `Package.swift` declares conditional dependencies via `.package(name: ..., condition: ...)`** (the SwiftPM platform-conditional dep API): v0.1 does NOT parse `Package.swift` at all (per FR-002), so conditional declarations are invisible to mikebom. The `Package.resolved` lockfile contains the resolved transitive set regardless of platform conditions, so the components still emerge in the SBOM; the conditional metadata is the deferred piece.
- **A `Package.resolved` from a build inside `.build/checkouts/`** (the SwiftPM build cache): mikebom MUST NOT traverse `.build/` per the milestone-114 / milestone-090 cache-exclude convention (the cache is a build artifact, not a source-tree-declaring file).
- **Two-deep nested Gradle workspaces** (a `settings.gradle.kts` `include(":foo")` where `foo/` has its own `settings.gradle.kts`): mikebom MUST treat only the OUTERMOST `settings.gradle.kts` as the workspace root. Inner `settings.gradle.kts` files are walked as `build.gradle.kts` candidates only — they don't create nested workspace roots.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: When mikebom encounters a `Package.resolved` file at any directory in the scan tree, it MUST parse it as JSON (with schema-version dispatch on the top-level `version` field) and emit one SBOM component per `pins[]` entry. PURL format: `pkg:swift/<host>/<namespace>/<name>@<version>` where `<host>` / `<namespace>` / `<name>` derive from the lockfile entry's `location` URL, stripping the `.git` suffix. The component carries `mikebom:source-files: "<path/to/Package.resolved>"` annotation.
- **FR-002**: `Package.swift` is NEVER parsed for dependency content in v0.1. Its presence is detected only as a signal that the containing directory is a SwiftPM project root (used to discover sibling `Package.resolved` files). When mikebom encounters a `Package.swift` without a sibling `Package.resolved`, it emits a `tracing::warn!` naming the unresolved manifest and emits ZERO components from that directory. Local-path declarations (`.package(path: "...")`), target declarations, conditional-platform deps, and workspace-member emission from `Package.swift` content are all deferred to a future phase — `Package.swift` is executable Swift code and parsing it safely requires either a Swift parser dependency or shelling out to the host's `swift` toolchain, neither of which v0.1 takes on.
- **FR-003**: For each `Package.resolved` entry whose `state.version` is absent but `state.revision` is present (SwiftPM commit-pinned mode), mikebom MUST emit the component with the FULL 40-char revision SHA as the PURL version segment (`pkg:swift/<host>/<ns>/<name>@<full-sha>`) and `mikebom:source-type: "git"` annotation. The same SHA is duplicated verbatim onto `mikebom:source-revision: "<sha>"` for grep-friendly cross-reference. The full-SHA-as-version convention matches the existing Go reader's `pkg:golang/...@<sha>` shape and preserves uniqueness without forcing consumers to re-encode.
- **FR-004**: When mikebom encounters a `build.gradle.kts` file at any directory in the scan tree, it MUST scan the `dependencies { ... }` block and emit one SBOM component per declared dependency. PURL format: `pkg:maven/<group>/<name>@<version>`. The component carries `mikebom:source-files: "<path/to/build.gradle.kts>"` annotation. Dependency-configuration → lifecycle-scope mapping per Acceptance Scenario US2#3. Tier: `mikebom:sbom-tier = "design"` — these components are gated by the existing `--include-declared-deps` flag (auto-on for `--path` scans, opt-in for `--image` scans) so artifact-tier faithfulness is preserved on image scans. When a `gradle.lockfile` (milestone 106) exists alongside `build.gradle.kts` and produces a component with the same canonical PURL, the lockfile-side entry wins on tier (`source` or `installed`) and the design-tier `build.gradle.kts` discovery is deduped away.
- **FR-005**: When mikebom encounters a `gradle/libs.versions.toml` file at the project root or a `gradle/` subdirectory, it MUST parse it as TOML and build a `(libraryAlias → group:name:version)` lookup table. References from `build.gradle.kts` to `libs.<alias>` MUST resolve through this table at component-emission time.
- **FR-006**: When a `build.gradle.kts` declares a Kotlin Multiplatform `kotlin { ... }` block with source-set dependencies, mikebom MUST emit each dep with `mikebom:kmp-source-set` annotation whose value is a JSON-encoded ARRAY of every source-set name that declared the dep (e.g., `"[\"commonMain\"]"` for a `commonMain`-only dep; `"[\"commonMain\",\"jvmMain\"]"` for one shared between source-sets, sorted lex for determinism). The kotlin_dsl reader MAY emit one `PackageDbEntry` per `(dep × source-set)` tuple pre-dedup AS LONG AS each duplicate carries the SAME merged source-set array under `mikebom:kmp-source-set`; the milestone-105 `scan_fs::dedup` pipeline collapses these into ONE canonical component post-emission, preserving the merged annotation. The post-dedup invariant ("exactly one component per canonical PURL") is the consumer-facing contract; the pre-dedup duplication is a reader-internal detail invisible to SBOM consumers. The JSON-array storage convention mirrors C64 `mikebom:produces-binaries` (milestone 116) and C67 `mikebom:assertion-conflict` (milestone 119).
- **FR-007**: When mikebom encounters a `settings.gradle.kts` file, it MUST treat its `include(":module")` declarations as workspace-member declarations and emit each module with `mikebom:component-role: "main-module"` consistent with the milestone-106 workspace-member convention. A synthetic workspace-root component derived from `rootProject.name` (or the directory name when absent) carries `mikebom:component-role: "workspace-root"` AND PURL `pkg:generic/<rootProject.name>@0.0.0` — matching the established uv / bun workspace-root convention. The `pkg:generic/` ecosystem is reused intentionally so cross-reader downstream consumers (e.g., the `--bind-to-source` flow) treat all multi-module workspace roots uniformly.
- **FR-008**: When BOTH a Kotlin/Gradle module AND a Swift Package coexist under one scan root (the KMP polyglot case from US3), mikebom MUST execute both readers via the existing `scan_fs::package_db::read_all` dispatcher composition without per-ecosystem coordination, and emit components for BOTH ecosystems in the same SBOM. PURLs in different ecosystems are distinct entities (no dedup collapse on name similarity); component count = Maven-side count + Swift-side count.
- **FR-009**: Parse failures on any individual `Package.resolved`, `Package.swift`, `build.gradle.kts`, `settings.gradle.kts`, or `libs.versions.toml` MUST emit `tracing::warn!` and yield zero components from that file. The walk continues on sibling files. Single-file parse failures NEVER abort the scan (mirrors the milestone-106 reader policy).
- **FR-010**: The two new readers MUST integrate via the existing `scan_fs/package_db/read_all` dispatcher in the same shape as `gradle::read` and `cargo::read`. Each reader gets its own module directory (`scan_fs/package_db/kotlin_dsl/`, `scan_fs/package_db/swift/`) consistent with the existing `gradle/` + `nuget/` + `pip/` layout.
- **FR-011**: Each new reader MUST honor the milestone-113 `--exclude-path` flag + the milestone-114 `safe_walk` invariants so operator-supplied directory exclusions are respected uniformly across every walker.
- **FR-012**: The `Package.resolved` reader MUST NOT make any network requests — it is a pure lockfile parser. Same constraint as every other lockfile reader (cargo, npm, pip, gem, gradle).
- **FR-013**: The new readers MUST NOT add any new runtime Cargo dependencies. Existing crates available: `serde` / `serde_json` (Package.resolved JSON parsing), `toml` (libs.versions.toml parsing), `regex` (build.gradle.kts pattern extraction — Kotlin DSL is a Kotlin expression language but the dep declarations follow well-defined surface syntax that regex can extract). No Kotlin or Swift parser dependency.
- **FR-014**: When mikebom emits a Swift-ecosystem component, it MUST honor the purl-spec's `pkg:swift/` canonical shape per the [package-url specification, swift type](https://github.com/package-url/purl-spec/blob/main/PURL-TYPES.rst#swift). Specifically: the `<host>/<namespace>/<name>` segments are mandatory; version (when known) appears after `@`; the `.git` suffix is stripped from the origin location before PURL projection.

### Key Entities

- **Swift Package Manifest**: A `Package.swift` Swift source file declaring the SwiftPM project structure. v0.1 detects its PRESENCE only as a signal that the containing directory is a SwiftPM project root; its contents are NEVER parsed. Dependency content flows exclusively through the sibling `Package.resolved` lockfile.
- **Swift Package Lockfile**: A `Package.resolved` JSON file recording version-pinned + commit-pinned dependencies. The primary discovery source for v0.1. Schema version (1 / 2 / 3) dispatched on the `version` integer at the top of the file.
- **Kotlin DSL Build Script**: A `build.gradle.kts` Kotlin source file declaring dependency requirements via a `dependencies { ... }` DSL. The primary discovery source for Kotlin / Android / KMP projects.
- **Gradle Version Catalog**: A `gradle/libs.versions.toml` TOML file declaring centralized version + library bindings. Resolved into a lookup table at parse time; references from `build.gradle.kts` to `libs.<alias>` consult it at emission time.
- **Gradle Settings Script**: A `settings.gradle.kts` Kotlin source file declaring multi-module workspace structure via `include(...)` calls. Drives workspace-member discovery and the synthetic workspace-root component.
- **KMP source-set**: A Kotlin Multiplatform source-set name (`commonMain`, `jvmMain`, `iosMain`, `androidMain`, `nativeMain`, …) attached to deps via the new `mikebom:kmp-source-set` annotation. The annotation value is a JSON-encoded array of every source-set that declared the dep (sorted lex for determinism), enabling downstream consumers to filter the SBOM to a single target via `JSON.parse(value).includes("<target>")` without component duplication.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: After this milestone ships, a developer scanning an Android-Studio-generated Kotlin DSL project sees every declared Maven dependency in the emitted SBOM. Today's behavior is an empty SBOM; the target is full coverage of all `dependencies { ... }` block entries in `build.gradle.kts` files across the project tree.
- **SC-002**: After this milestone ships, a developer scanning a Swift Package Manager project sees every entry from `Package.resolved` as a `pkg:swift/...` SBOM component with the resolved version + commit-pinned source location preserved. Today's behavior is an empty SBOM.
- **SC-003**: A Kotlin Multiplatform polyglot monorepo scanned at the repo root produces one SBOM containing BOTH `pkg:maven/...` (Android side) AND `pkg:swift/...` (iOS side) components, plus dependency edges from workspace-root to each module preserved.
- **SC-004**: A `sbom scan --path <root>` against a Kotlin / Android project with no `gradle.lockfile` (the default project-template state) produces a non-empty SBOM by default. Today's behavior is empty per the milestone-106 reader's lockfile-only discovery; the target is graceful coverage from `build.gradle.kts` + `libs.versions.toml` alone via the design-tier path (`--path` auto-enables `--include-declared-deps`; image scans of the same source tree stay artifact-tier-only unless the operator opts in).
- **SC-005**: A `Package.resolved` schema-version mismatch (e.g., a project using SwiftPM 6.0+ that emits `version: 4`) produces a warn-and-continue with a clear log line naming the file + version; the scan does NOT abort. No partial SBOM is ever emitted with mangled data.
- **SC-006**: The new readers add ZERO new Cargo dependencies to `mikebom-cli/Cargo.toml`. The dependency-tree-at-lockfile-level is unchanged from milestone 121's baseline.
- **SC-007**: When BOTH readers' walk sites are observed for a scan with no Kotlin or Swift content (e.g., a pure Cargo project), the scan output is byte-identical to a pre-feature mikebom build. The new readers contribute zero components and zero annotations when nothing matches; no regression on existing ecosystems.

## Assumptions

- The `Package.resolved` discovery-first design (US1) follows the cargo `Cargo.lock`-first / gem `Gemfile.lock`-first pattern. `Package.swift` content (manifest-declared deps, local-path / workspace-member emission, target declarations, conditional-platform deps) is deferred to a future phase — `Package.swift` is executable Swift code, and safe parsing requires either a Swift parser dependency or shelling out to the host `swift` toolchain. The lockfile-only path covers the dominant operator workflow.
- The `build.gradle.kts` parser is regex-based, not a full Kotlin parser. Kotlin DSL's dep declarations follow a well-defined surface syntax (`implementation("group:name:version")`, `implementation(libs.alias)`, `implementation(group = "...", name = "...", version = "...")`) that regex extraction covers. Operators writing exotic dep declarations (heavy meta-programming, custom DSLs, deps declared via reflection) get a `tracing::warn!` and a degraded SBOM — the documented contract is "common surface syntax, not arbitrary Kotlin."
- Kotlin Multiplatform source-set extraction (FR-006) annotates source-set names but does NOT translate them into per-platform SBOMs. v0.1 emits a single SBOM with `mikebom:kmp-source-set` annotations; consumers wanting per-platform SBOMs filter post-hoc. Per-platform SBOM emission is deferred.
- The PURL for a Swift Package uses the [purl-spec swift type](https://github.com/package-url/purl-spec/blob/main/PURL-TYPES.rst#swift) which has stable type definition + examples. mikebom emits the canonical shape; consumers (deps.dev, advisory databases) that don't yet support `pkg:swift/` see the component but cannot enrich it — same as any new-ish ecosystem.
- Per the milestone-002 fail-closed principle, parse failures on individual files emit `tracing::warn!` and yield zero components for that file but do NOT abort the scan. The walk continues on sibling files.
- Cross-platform: the readers MUST work on macOS (the dominant Swift dev platform), Linux (Android Studio + Kotlin server-side dev), AND Windows (Kotlin Multiplatform's Windows target — a small but growing cohort). No `#[cfg(unix)]` gating; tests assert behavior parity across all three runners per the milestone-100 Windows CI lane.
- No new parity-catalog rows (C68+) are introduced. The annotations this milestone surfaces — `mikebom:source-files` (existing C18), `mikebom:source-type` (existing C1), `mikebom:component-role` (existing C40), `mikebom:lifecycle-scope` (existing C42), `mikebom:kmp-source-set` (NEW) — reuse existing rows or add ONE new row for KMP. The KMP row's Principle V audit is straightforward (no native CDX / SPDX 2.3 / SPDX 3 field for "which Kotlin source-set declared this dep").
- The milestone-114 `safe_walk` invariants + the milestone-113 `--exclude-path` flag continue to apply uniformly. No new walker code paths bypass either.
- A future milestone-122-phase-2 may add CocoaPods + Carthage support for the Apple ecosystem if operator demand surfaces. v0.1 covers SwiftPM only because it's the modern Apple-blessed dependency manager and CocoaPods adoption is declining.
